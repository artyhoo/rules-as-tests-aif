## §1 Why this kickoff exists

**The risk this kickoff mitigates:**

The maintainer is a single operator. Between sessions, context lives in three places: (a) git history + drafts on disk, (b) memory entries, (c) maintainer's own mental model. Item (c) decays naturally over weeks. Items (a) and (b) survive but require re-reading.

The goal-clarity-dialogue 1A session (2026-05-16) produced:
- 8 drafts in `goal-clarity-dialogue/drafts/`
- 1 research-patch in `docs/meta-factory/research-patches/`
- 2 new kickoff folders (`prose-rules-audit-research/`, this one)
- 3 memory entry updates
- 1 atomic-commit-plan describing 8 commits + 2 R-phase sessions + 1 verification + 1 deadline

This is **significant queued work** spread across 2-4 weeks. Without a coordinator perspective, risks:

1. **Critical-path inversion** — shipping Commit 3 before prose-rules R-phase = shipping unverified theatre
2. **Time-sensitive items missed** — Commit 6 has explicit ≤2-week deadline; if forgotten → fallback to violation-rate promotion (still OK, but worth knowing)
3. **Self-reinforcing drift** — session-bootstrap-hook auto-injects narrow «fails CI» framing until Commit 1 ships. Every session-day pre-Commit-1 = one more session that pattern-matches on narrow framing.
4. **Recursive antipattern** — 1A caught `#discipline-theatre` in prose-rules audit. Same antipattern could recur if next session ships Commits 3/4/5 without prose-rules R-phase running first.
5. **Work-without-goal drift** — if maintainer triggers a session («продолжай») without specifying which commit/R-phase to do, AI defaults to plausible-looking work — possibly wrong critical-path direction.

**Orchestrator role:** keep all five risks visible. Surface critical path. Prevent silent skipping. Maintain coherence across sessions.

### §3.5 Self-reinforcing drift (passive risk)

Session-bootstrap-hook auto-injects «fails CI on violation» at every prompt submit. Until Commit 1 ships, every new session sees narrow framing first. Time-sensitivity:

- **1 week pre-Commit-1:** acceptable, low cumulative drift
- **2-3 weeks pre-Commit-1:** moderate drift risk — sessions internalize narrow framing as anchor
- **4+ weeks pre-Commit-1:** drift becomes self-reinforcing — Commit 1 itself may face «is this actually needed?» pushback from sessions running under narrow anchor

**Mitigation:** Commit 1 is the HIGHEST priority for maintainer to ship. Independent of any research session. 30-40 min one-shot.

## §6 Orchestrator methodology

### §6.1 First-action decision tree

When this session starts (orchestrator mode), follow this tree:

```text
Maintainer's first message contains specific request?
├── YES: "apply commit N" / "do R-phase" / "ship X"
│        → Skip to §6.2 Execution path
└── NO: open question or "what's next?"
         → Skip to §6.3 Status path
```

### §6.2 Execution path (specific request)

1. **Match request to inventory** (§3) — exact identifier (Commit N, R-phase name, verification step)
2. **Check blockers** — is the requested item blocked? If yes, surface blocker + alternative
3. **Check time-sensitivity** — does the request conflict with a higher-priority anti-drift item (e.g. Commit 1 still not shipped 3 weeks in)? Surface tradeoff
4. **Choose pathway:**
   - Small atomic commit (Commits 1, 2, 3, 4b, 7) → orchestrator session can apply directly via Edit/Write/Bash on drafts (or delegate to fresh Sonnet for clean session)
   - Large focused session (R-phase, Commit 6) → recommend separate fresh Opus or Sonnet session per orchestrator skill «делегировать через Agent tool»
5. **§1.7 Forward+Backward check** before close — does this commit/action comply with all active disciplines? Sweep applied?
6. **Status update** to memory or research-patch on close — preserves continuity across sessions

### §6.3 Status path (open question)

1. **Surface inventory snapshot** (§3.1 + §3.2 worklist with current state)
2. **Highlight time-sensitive items**: Commit 1 anti-drift window; Commit 6 deadline (if Commit 2 already shipped)
3. **Highlight blockers**: Commits 3/4/5 still need prose-rules R-phase; Commit 7 needs alt-target research
4. **Surface decision points**:
   - Maintainer ready for Commit 1?
   - Maintainer has 3-4h focused for R-phase soon?
   - Wave 10 R-phase planned — Commit 4b ready to land?
5. **Wait for direction** — do NOT autonomously pick «most important» work; offer ranked options + tradeoffs

### §6.4 Delegation strategy (per orchestrator skill)

Per `~/.claude/skills/orchestrator/SKILL.md` discipline:

- **Старшая (orchestrator session — likely Opus):** plans, surfaces options, delegates, accepts via evidence
- **Младшая (focused execution session — Sonnet):** applies a specific commit/R-phase per its kickoff

**Delegate to fresh Sonnet when:**
- Multi-file atomic commit (Commit 1: 3 files — clean session benefits)
- R-phase 3-4h focused work (Sonnet preferred for execution discipline)
- Single targeted edit (Commit 3/4b: ~15 min, also fine for Sonnet)
- Worktree-isolated parallel work (if maintainer ever runs multiple Commits in parallel — REQUIRED per `parallel-subwave-isolation.md`)

**Keep in orchestrator session when:**
- Quick status check, plan adjustment, surface risks
- Decision points requiring tradeoff articulation
- WebSearch verification (15 min — efficient to do in same session)

Every orchestration session closes with explicit:
- What changed in inventory (commits shipped, R-phase complete, etc.)

**Pre-ship review is the close of that obligation.**

**Mandatory process (per draft, before applying as commit):**

1. **Read the draft fully** — not skim
2. **§1.7 Forward-check** against the draft as a discipline-bearing change:
   - Does the change comply with all active layers (R1-R20, principles 01-09, capability-commit gate, build-vs-reuse SSOT, trigger sweep, doc-authority headers)?
3. **§1.7 Backward-check:**
   - Does the draft's recommendation apply complete sweep of existing artefacts under its scope?
   - Exemption mechanism explicit (where applicable)?
4. **Identify every VERIFIABLE claim in the draft** and probe each:

5. **Identify «Self-application» / «T15» / «Self-review» blocks in the draft** and audit them substantively — does the claim demonstrate HOW (substantive) or only CHECK (syntactic-only theatre)?
6. **Verdict per draft:**
   - ✅ **PASS** — draft ships as scheduled
   - ⚠️ **REVISE** — specific gaps identified; ship after corrections
   - ❌ **BLOCK** — fundamental issue; refer back to dialogue/research session
7. **Output one consolidated self-review patch:** `docs/meta-factory/research-patches/2026-MM-DD-1a-drafts-substantive-review.md` per T7 template (Problem → Root Cause → Solution → Prevention → Tags). One patch for all 8 drafts (coherent self-review unit; 1A as common origin).
