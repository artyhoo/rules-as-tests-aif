## §3 Stage gates (real git checks — NOT in-memory FIFO)

Each gate is a real `gh pr list --search "is:merged ..."` check. A stage's dispatch is blocked until its dependency PR is merged to `staging`.

### S1a / S1b → S2

```bash
gh pr list --search "is:merged head:<s1a-branch> base:staging" --json number,mergedAt --limit 5
gh pr list --search "is:merged head:<s1b-branch> base:staging" --json number,mergedAt --limit 5
```

Both non-empty → S2 unblocked. Either empty → HALT S2.

### S2 → S3 → S4 → S5

Each downstream stage's dispatch waits for the prior stage PR to merge to `staging` (same `gh pr list --search "is:merged head:<prev-branch> base:staging"` pattern). Empty → HALT. **Phase -1 cold-review between every stage transition is MANDATORY** (see §7).

## §4 Per-stage dispatch scope

Each stage is dispatched to aif via a **stage-scoped /tmp kickoff** (§4c), NOT this multi-stage doc. The /tmp kickoff carries: the focused build steps, acceptance-by-command-output, the §4c park-don't-guess contract, the §4b §1.7 PR-body mandate, and the §5 traps.

- **S1a (ESLint forbid pilot):** take ONE simple `forbid` principle (single syntactic selector — e.g. forbid `.only` in test files, or another simple forbid not already in `eslint-rules/`), run it through the pipeline by hand on `no-restricted-syntax`: spec → rule data → paired good/bad fixtures → run via `RuleTester` → capture red(bad)/green(good) **output**. Produce a markdown reference recording the pipeline points (where spec lives, where data lives, where fixtures live, how it runs) so S2–S5 have fixed anchors. Acceptance: bad reddens + good greens by actual run (output in the reference).
- **S1b (ast-grep composite pilot):** take ONE composite/relational `forbid` (`not`+`has`/`inside`, no types — e.g. "forbid an empty catch block", or "`expect` call not inside an it/test block") on ast-grep YAML; attempt the same in `no-restricted-syntax`; show **on the facts** where ast-grep's composite rule expresses what the flat esquery selector cannot (or does poorly). Verdict with example/numbers: for which class is ast-grep the engine of choice. **If ast-grep is NOT better — record that and do NOT adopt the dependency.**
- **S2 (declarative type + engine seam):** `check.type:"declarative"` in `recipe.schema.json` + `types.ts` with `engine?: "eslint-restricted" | "ast-grep"` (optional, default `eslint-restricted`; selector, messageId, `presence:"forbid"`) + **eslint-restricted runner + 1 eslint-restricted test recipe NOW** (4 wiring points: `gate-rule-tester.ts:69/:80/:87`, `gate-schema.ts:23`, `synthesize.ts:96`) + committed gap-rationale (esquery over-flags metavar, reproduced). **ast-grep = reserved schema value; runner/dep/recipe DEFERRED per decision (i)** — `engine:"ast-grep"` yields an explicit deferred-marker, not a silent pass. Existing 4 check types unbroken (snapshot green). **No new dependency in S2** (ast-grep NOT added).
- **S3 (anti-vacuity validator, engine-agnostic):** autofix-clean + messageId-coverage + single-token-diff assert; snapshot gate. A rule of ANY engine is rejected on: breaking autofix / unreachable messageId / good=bad on >1 token. Each gate ships its own paired-negative fixture on both engines. Deterministic, zero LLM.
- **S4 (forbid compilation — moat):** `synthesize` **compiles** a `forbid` spec → rule data on the engine chosen by rule shape + paired fixtures + provenance header; generator is the sole writer. New forbid = adding a **data spec** (with engine) WITHOUT a handwritten rule file and WITHOUT a new in-box template; emit passes S3 gates; run on ≥1 forbid of each justified engine (output in PR). Moat (1–2 wk) — likely starts, won't finish overnight.
- **S5 (anti-hand-edit gate):** provenance-hash/generated-marker + CI check: manual edit of an emitted file → red. Mechanical failure (a run rejection), not "should".
