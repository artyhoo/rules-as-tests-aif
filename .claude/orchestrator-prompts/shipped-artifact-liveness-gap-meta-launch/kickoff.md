## §5 AI-traps active (per `.claude/rules/ai-laziness-traps.md §3`)

See `.claude/rules/ai-laziness-traps.md §2` for the full canonical catalogue.

**Active canonical traps for shipped-artifact-liveness-gap** (carried forward from the umbrella kickoff §AI-traps and re-justified):

- **T1** — «looked at 3 examples, all clean, category done»: the agent/skill/hook population is small; sample the FULL set of shipped capabilities, not the convenient few.
- **T2** — designing the methodology ≠ running it: the R-phase MUST actually *dispatch* a shipped agent (or describe the concrete dispatch + observed `tool_uses`), not merely propose «a probe would catch this».
- **T11** — designing a custom gate/probe without prior-art check: run the BFR-default search before any «I propose a gate».
- **T12** — skipping the literature sweep because «I know liveness testing»: WebSearch ≥3 phrasings at proposal-time, not from training memory.
- **T14** — conflating «no findings» with «no theatre»: a clean form-check at low behavioural coverage = «coverage insufficient to conclude», not «agents live».
- **T15** — self-application mandatory: does the proposed mechanism test ITSELF (would it have caught #551)? The R-phase acceptance requires this note explicitly.
- **T16** — pattern-matching-on-name: a check named «validate» that does not run the artifact is exactly the #551 root cause; distinguish form-check from behaviour-check.

**Domain-specific trap (shipped-artifact-liveness-gap-specific, NOT in canonical catalogue):**

- **T-LIVENESS-A** — «proposing a `tools:`-name-allowlist gate and calling liveness *solved*». A valid-name `tools:` entry still does not prove the agent *uses* the tools on dispatch. The R-phase MUST distinguish the cheap form-check (name ∈ allow-list) from the behaviour-check (dispatched agent actually invokes its tools / gate actually fires) and not let the former masquerade as the latter.
