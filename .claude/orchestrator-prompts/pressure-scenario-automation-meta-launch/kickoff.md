## §5 AI-traps active (per `ai-laziness-traps.md §3`)

See `.claude/rules/ai-laziness-traps.md §2` for full catalogue.

**Active canonical traps for pressure-scenario-automation:**

- **T2** (designing-≠-running) — designing a generator does NOT prove it generates *strong* traps. The R-phase MUST run an actual RED→GREEN dispatch on generated output, not describe one.
- **T11 / T12** (search-before-build / no-skip-literature-sweep) — BFR-default search (DeepWiki + WebSearch ≥3 phrasings) for existing scenario-generators BEFORE any "I propose a generator". Training-data knowledge of SP `writing-skills` is not a substitute for a present-moment sweep.
- **T6** (no self-reported high-confidence) — quantify scenario strength by actual RED→GREEN pass-rate, not the word "high".
- **T16** (pattern-matching-on-name) — a generator that emits plausible-*looking* scenarios that don't actually pressure. Upstream problem-class match must be verified, not assumed (SP `writing-skills` pressure-scenarios vs ours).

**Domain-specific traps (pressure-scenario-automation-specific, NOT in canonical catalogue):**

- **T-PSA-A** (carried from umbrella kickoff `T-genscenario-A`) — «accept a generated scenario because it *reads* adversarial». Every generated scenario MUST be auto-validated by an actual RED→GREEN dispatch (baseline must FAIL) before it is trusted. An unvalidated generated trap is exactly the weak-trap false-negative this track exists to prevent.
- **T-PSA-B** (meta-orchestration-specific) — «pre-commit the spec/plan/impl stages because the pipeline *names* them». The umbrella's acceptance is the R-phase verdict; treating Stages 2–4 as decided before the falsifier resolves is `#strategy-decided-by-reviewer` (reviewer-discipline §3) smuggled into a launch-table.
