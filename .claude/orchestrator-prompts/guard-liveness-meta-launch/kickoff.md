## §3 Stage gates (real git checks — NOT in-memory FIFO)

### Stage 1 → Stage 2

Stage 2 (v1.5 ‖ v3) dispatch waits for v0 to be merged to `staging` (v1.5/v3 inherit v0's criticality table). Branch names are Worker-chosen → gate on slug-in-title + base:

```bash
gh pr list --state merged --search "guard-liveness-v0-audit in:title base:staging" \
  --json number,title,mergedAt,headRefName --limit 5 2>/dev/null \
  || echo "gh unavailable — cannot verify Stage 1 gate"
```

Expected: ≥1 merged PR carrying the v0 audit (the criticality research-patch landed). If empty → HALT. Do NOT dispatch v1.5/v3.

> Note: v1 (`guard-liveness-gate`) is also Stage 1 but is NOT a Stage-2 prerequisite — only v0's table gates v1.5/v3. v1 must land before Stage 3 (v2 depends on v1).

### Stage 2 → Stage 3

Stage 3 (v2) dispatch waits for **v1 AND v1.5 AND v3** ALL merged to `staging`:

```bash
for slug in guard-liveness-gate guard-liveness-v1.5-cmd-script guard-liveness-v3-manual-sp; do
  n=$(gh pr list --state merged --search "$slug in:title base:staging" --json number --jq 'length' 2>/dev/null)
  echo "$slug: ${n:-0} merged PR(s)"
done
```

Expected: all three report ≥1. If any reports 0 → HALT. Do NOT dispatch v2.

**Phase -1 cold-review between every stage transition is MANDATORY** (see §7 below).
