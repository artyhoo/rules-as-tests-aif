## §1 Цель (одной фразой)

На чистом проекте, на **актуальной** Node (включая 22), подброшенное нарушение (`as any`, плавающий промис, импорт из запретного слоя) **реально блокируется** на самом раннем достижимом канале (edit-time ESLint → pre-commit → pre-push); «зелёно» больше не врёт.

## §2 Зафиксированные решения (не пересматривать без причины)

1. **Корень бага — загрузка `.ts`-правил у консьюмера.** Шипнутый `templates/*/eslint.config.mjs` импортит барель как `.ts` (`./eslint-rules-local/index.ts`), который импортит сиблинг-правила `.ts`. ESLint консьюмера должен грузить TypeScript на лету → требуется TS-загрузчик → ломается на Node 22 (нативный strip-types меняет поведение импорта `.ts`). Тест `f17-lint-rules-planted-violation` краснеет на Node 22, зелёный на Node 20 (тот же класс, что чинили #636/#642/#644). База: `plans-check.md` §5/§6 (operator notes, вне репо).
2. **РЕШЕНИЕ — вариант A (operator 2026-06-26): компилировать правила в JS + `.d.ts` при установке.** Авторинг правил остаётся на TS; `setup.d/40-configs.sh` при установке генерит **скомпилированный** `eslint-rules-local/index.js` + типы, шаблон `eslint.config.mjs` импортит `./eslint-rules-local/index.js`. Это убирает зависимость от TS-загрузчика и версии Node совсем — ровно так шипит сам TypeScript-тулинг (автор на TS, поставка скомпилирована). **Формат скомпилированного бареля (CJS либо ESM) пинить явно в одном месте и держать импорт шаблона (`eslint.config.mjs`) согласованным с ним** — деталь S2, но зафиксировать, чтобы гейт §7 прогонял ровно тот путь, которым барель грузит консьюмер (см. §7 S2-gate: проверка через ESM-`import`, не CJS-`require`).
3. **Вариант B (отдавать `.ts` + держать загрузчик) — ОТКЛОНЁН.** Перекладывает хрупкость version-loader'а на каждую новую Node консьюмера; гибрид (по версии Node разный формат) — ещё хуже (двойной путь + детект, рассинхронящийся при апгрейде Node). JS работает на всех Node, поэтому ветка под TS не даёт функциональной выгоды.
4. **Зелёно не должно врать.** Тихий REDUCED-режим (`pre-push.fallback.sh`), когда полные проверки не запускаются, обязан быть **видимым предупреждением**, а не печатать «passed».
5. **CI-честность.** Сейчас CI = Node 20 (Node 22 в матрице нет — отсюда баг и проскочил). Node 22 добавляется в матрицу как явный deliverable.

## §4 S1 — детальная задача (go сейчас)

На Node 22, в чистом проекте после `bash /path/setup ts-server` + dev-deps:
1. Создать файл с заведомым нарушением: `const x = y as any;`.
2. Запустить ровно консьюмерскую команду (`npx eslint <file>`) и зафиксировать **не** красный/зелёный, а **загрузку плагина**: грузится ли `eslint-rules-local/index.ts` без ошибки. Ожидаемая разводка причин: (а) «Unknown file extension ".ts"» / ошибка загрузки → нет TS-загрузчика при eslint-ране; (б) плагин грузится, но правило не сработало → другая причина.
3. **Выход S1:** воспроизведение + точная причина (команда + вывод). Ничего не чиним.

## §7 Гейты стадий (реальные проверки, не «vibe»)

- **S2 gate:** `node --input-type=module -e "await import('./eslint-rules-local/index.js')"` грузится без ошибки — тот же ESM-`import`, что делает шаблонный `eslint.config.mjs`, так гейт прогоняет **реальный путь потребителя**. (Не CJS-`require`: его результат на ESM-бареле **зависит от версии Node** — `ERR_REQUIRE_ESM` на Node 20 и ранних 22.x; проходит на ≥22.12, где `require(esm)` разрешён для модулей без top-level await — ровно та version-хрупкость, которую U2 искореняет; `import` стабилен по версиям и грузит барель и в CJS-, и в ESM-форме — проверено эмпирически 2026-06-26, Node 24.) На чистом проекте, Node 22, `npx eslint <file с as any>` **падает** по правилу. Барель — `.js`, рядом `.d.ts`.
- **S3 gate:** на не-`src/` раскладке (`app/`, плоское дерево) аудит даёт WARN/SKIP (не PASS), либо реально итерирует файлы — grep по `probeR4`/`R17`, прогон на фикстуре не-`src/`.
- **S4 gate:** при отсутствии dev-deps `pre-push` печатает явный WARN про REDUCED (не «✅ passed»).
- **S5 gate (умбрелла-закрытие):** CI-workflow содержит Node `20` и `22` в матрице; `f17-lint-rules-planted-violation` зелёный на обеих; `done.md` с этим доказательством.
