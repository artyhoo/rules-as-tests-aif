### §1.2 Three concrete problems (per kickoff §1, verified)

1. **C3 shell tests exit-code-only** — `test_D1` (`audit-ai-docs.test.sh:148-159`) and `test_D3` (`audit-ai-docs.test.sh:183-197`) check exit code + one grep pattern per Wave 9 M-grade fix. 2. **C4 false negatives from regex** — `audit-ai-docs.sh` D-probes use `grep -qF` (literal string matching) and `awk` on AGENTS.md skill declarations. These miss structural variants:
   - D1 uses `awk 'match($0, /skill `[^`]+`/)` — misses alternate declaration formats
   - D3/D5 use `grep -qF "$CANON_PHRASE"` — literal string match, no synonym tree beyond `$CANON_ALT`
   - No AST-level checking of any TS source file in D-probes

3. **Stryker zero mutation coverage** — `.husky/pre-push` (468 lines) and `audit-ai-docs.sh` (311 lines) have zero Stryker coverage. `stryker.config.mjs` mutates `eslint-rules/**/*.ts` only. Shell logic (string comparisons, regex patterns, exit code routing) cannot be mutated by StrykerJS.

### §1.3 Root cause (confirmed)

`#own-stack-blind-spot` — TypeScript (`tsx ^4.19.0`) was in `packages/core/package.json` from the beginning but was never considered for hooks. Each R-phase had bash hooks as inert infrastructure outside its scope.

## §4 Prior art findings

All candidates evaluated via context7 AND deepwiki (per kickoff §5.-1 dual-channel mandate).

### §4.1 Husky (SSOT: existing, not yet entered)

**Upstream problem class:** Git hooks made easy — shell script dispatch. Zero-config activation via `prepare` script.
**Our problem class:** Testable, Stryker-mutatable, AST-capable hook logic.

**T16 match check:** Husky problem class = hook *activation* (setting `core.hooksPath`). Our problem class = hook *logic quality* (testability, mutation coverage). **MISMATCH on the testability dimension** — Husky deliberately stays minimal and delegates logic to user scripts. Husky does not solve "how to test hook logic" — that falls to whatever runs inside the hook.

**context7 finding** (`/typicode/husky`): Pattern for non-shell hooks: shell entrypoint file dispatches to `node .husky/pre-push.js`. No built-in TypeScript support — user adds `node --import tsx/esm` manually. Confirmed: Husky is orchestration only, not a logic testability framework.

**deepwiki finding** (typicode/husky): "No TypeScript-specific architecture... The responsibility for language support falls on the user." "Hook scripts should be POSIX-compliant shell scripts for maximum compatibility." Confirmed: Husky itself stays language-agnostic.

**Conclusion:** Keep Husky as activation layer (sets `core.hooksPath`). The 10-line bash bootstrap in `.husky/pre-push` that dispatches to `node --import tsx/esm packages/core/hooks/pre-push.ts` is the correct pattern. Replacing Husky with Lefthook or simple-git-hooks does NOT solve the testability problem.

**SSOT:** No new entry needed — Husky is prior art for orchestration (WATCHLIST status appropriate). Existing Wave 7 reference sufficient.

### §4.2 Lefthook (new SSOT candidate)

**Upstream problem class:** Fast parallel Git hooks manager for any project type. YAML config, binary install (Go), multi-language `runner:` support.
**Our problem class:** Same testability requirement as above + multi-stack support (consumer projects may be Go/Python/Ruby).

**T16 match check:** Lefthook's runner-agnostic model is a genuine partial match — `runner: node` or `runner: tsx` in `lefthook.yml` allows TS execution. However:
- Lefthook's "single YAML config" model conflicts with our "per-stack detection + fallback" requirement
- Lefthook does not add testability: the logic inside the hook (trailer parsing, LOC counting) still cannot be unit-tested or Stryker-mutated by Lefthook itself
- **MISMATCH on testability** — same as Husky: Lefthook = orchestration layer, not logic-quality layer
- **Partial match on multi-stack** — Lefthook handles Go/Python/Ruby via binary runner, which reduces the need for our bash-fallback. But project requires consumer stack independence (install.sh), which Lefthook doesn't address.

**context7 finding** (`/evilmartians/lefthook`): `runner: node` for JS scripts; `runner: bash/python/go` for other languages. Parallel execution via `parallel: true`. Configuration via `lefthook.yml` only — no programmatic API.

**deepwiki finding** (evilmartians/lefthook): "Lefthook does not natively support TypeScript files directly... you can run Node.js/tsx commands through its flexible command and script execution system." Scripts go in `.lefthook/{hook-name}/` with `runner: tsx`. No intrinsic testability or mutation support.

**Architecture implication for D1:** Migrating from Husky to Lefthook is a horizontal move (same orchestration layer, different format). Does not unlock testability. Would require replacing `.husky/` with `.lefthook/` directory — non-trivial refactor for zero testability gain. **DEFER Lefthook adoption** per T-Wave10-A (TS-core must be full, bash-fallback minimal — Lefthook doesn't help with either).

**SSOT entry proposed: #47** (see §12).

### §4.3 simple-git-hooks (new SSOT candidate)

**Upstream problem class:** Zero-dependency minimal git hooks via `package.json` config. Single command per hook.
**Our problem class:** Same testability + multi-check orchestration.

**T16 match check:** STRONG MISMATCH.
- simple-git-hooks: one command per hook. Our pre-push has 8 active sections. Would require a bash script wrapper anyway.
- No testability, no multi-runner, no stack detection.
- Design philosophy: "simple configuration in package.json" — antithetical to our complex hook logic.

**context7 finding** (`/toplenboren/simple-git-hooks`): Zero-dependency, 10.9kB. `package.json` `"pre-push": "npx tsx ./hooks/pre-push.ts"` format works — but that's exactly our own-build pattern.

**deepwiki finding** (toplenboren/simple-git-hooks): "Only one command per Git hook." "Requires manual execution of `npx simple-git-hooks` every time you change hook commands." Static scripts written to `.git/hooks/`. No parallel execution, no conditional logic, no file-pattern filtering.

**Verdict: REJECT** for this use case (not general REJECT). Project requires 8 checks in pre-push, stack detection, bash-fallback path. simple-git-hooks cannot express this without a wrapper bash script — which is exactly what we already have.

**SSOT entry proposed: #48** (see §12).

### §4.4 Danger JS as runtime engine (SSOT #41 ADOPT — re-evaluating fit-for-purpose)

**Upstream problem class:** Automated PR-body code review automation — validates PR descriptions, file changes, commit messages against team norms. Posts feedback to GitHub/GitLab/Bitbucket PRs.
**Our problem class for D6 (engine candidate):** Full hook runtime engine — must cover ALL 9 sections of `.husky/pre-push` in `danger local` mode.

**T13 check (ADOPTED mechanism, verify problem-class match):**

Danger JS is ADOPT in SSOT #41. The question for D6 is: does Danger's API cover all 9 pre-push sections?

**context7 finding** (`/danger/danger-js` + `/websites/danger_systems_js`):

## §5 Architecture decision

### §5.1 D6 three-branch analysis

**Branch (a): Own-build TS hook runner**
```text
.husky/pre-push (10 lines bash)
  → node --import tsx/esm packages/core/hooks/pre-push.ts
```

- **Pro:** Full control over API design. All logic testable. Stryker-mutatable. No new dep (tsx already in stack). Fallback path: if `node` not found, bootstrap bash handles critical-only checks.
- **Con:** We implement ourselves what Danger's `danger.git` already provides for commit/diff parsing (§7, §8). Duplicates Danger API for commit message access.
- **Stryker compatibility:** 100% — all TS logic in `pre-push.ts` is mutatable by existing StrykerJS config.
- **Mutation scope:** `packages/core/hooks/**/*.ts` added to `stryker.config.mjs` `mutate` array.

**Branch (b): Danger as full runtime engine**
```text
.husky/pre-push (10 lines bash)
  → npx danger local --dangerfile packages/core/dangerfiles/lite.ts
```

- **Pro:** Danger's `danger.git.commits[].message` natively covers §7 + §8 trailer checks. Shared module with PR-time (`danger ci`). SSOT #41 ADOPT.
- **Con:**
  1. Sections §1-§6 require `child_process` shell-outs from dangerfile — Danger adds no API value for actionlint, zizmor, npm scripts
  2. Danger was designed for CI post-merge, not pre-push enforcement — `danger local` is a secondary use case with known limitations (`danger.github` unavailable)
  3. `danger local` requires `--base main` or `--base master` — brittle if branch name differs
  4. LOC detection for capability commits requires `diffForFile()` per-file API (N calls vs. 1 `git diff-tree` call) — performance regression
  5. **`danger local` output mechanism:** `fail()` posts GitHub PR comment — in local mode, outputs to stdout. Error propagation via `process.exit(1)` only if dangerfile throws — `fail()` calls do NOT automatically exit non-zero. Requires explicit `throw` or `process.exit()` after failures.
  6. No bash-fallback path for non-TS stacks — Danger is a JS tool; non-Node stacks need wrapper anyway
  7. Dependency: requires `danger` as explicit dep — this IS a capability commit (dep in package.json)

**Branch (c): Hybrid**
```text
.husky/pre-push (10 lines bash)
  → node --import tsx/esm packages/core/hooks/pre-push.ts
  pre-push.ts:
    - §1-§6, §8/lychee: direct execSync() calls
    - §7 Prior-art trailer: own-build TS (TRIVIAL-REGEX-PORT)
    - §8 §1.7 trailer: own-build TS (TRIVIAL-REGEX-PORT)
  .github/workflows/discipline-self-check.yml → danger ci (D7 Wave 11)
```

- **Pro:** TS core owns all logic. No Danger dep for pre-push. Stryker covers everything. Danger deferred to Wave 11 for PR-time only (where its API shines: `danger.github.pr.body`).
- **Con:** Danger adoption delayed for PR-time surface.

### §5.2 RECOMMENDATION (D6)

**Recommend Branch (a) own-build TS hook runner, with Danger deferred to Wave 11 for PR-time only.**

**Rationale:**

1. **T13 problem-class match:** Danger's primary problem class is PR-body review automation with GitHub API integration. Our pre-push primary problem class is commit-message trailer validation + tool invocation. The commit-message parsing (§7, §8) is TRIVIAL-REGEX-PORT — 30-50 lines of TS, no external library needed. Using Danger as the engine for pre-push adds a 13.x-stable dependency to solve a problem that 50 lines of TS solves natively.

2. **Stryker-mutability (T15 self-application):** Own-build TS hooks in `packages/core/hooks/pre-push.ts` are 100% Stryker-mutatable by extending the existing `stryker.config.mjs` `mutate` array. Danger's dangerfile API (`fail()`, `warn()`) is external — Stryker would mutate the *conditions* feeding `fail()` but not the Danger framework internals. Both options are roughly equivalent on mutation coverage of our logic.

3. **Process.exit() semantics:** Own-build TS hook: `process.exit(1)` on failure — clear semantics, standard Node.js. Danger: `fail()` doesn't exit, requires explicit throw or `danger --fail-on-errors` flag. This is a footgun in pre-push: if a Danger internal error occurs without `fail()` firing, the hook may pass silently.

4. **No new dep (capability commit avoidance):** tsx is already in stack. Own-build TS hook runner = zero new deps. Danger as pre-push engine = new explicit dep in `package.json` = capability commit requiring Prior-art trailer.

5. **Fallback path:** Own-build allows: `if command -v node; then node --import tsx/esm pre-push.ts; else bash pre-push.fallback.sh; fi`. Danger requires Node anyway — same fallback needed.

6. **D7 implication:** If D6=(a), PR-time surface stays in Wave 11. Danger's ADOPT verdict (SSOT #41) is best exercised at the PR-time surface where `danger.github.pr.body` is available and Danger's PR-feedback-posting API is its core strength, not a workaround.

**Risk:** Implementing `pa_is_new_dep_added()` and `s17_check_trailer()` in TypeScript introduces risk of regression if the TS port misses an edge case. Mitigation: paired-negative arm in same commit (Wave 8 D6=C precedent). Stryker run after each sub-wave.

**Build-vs-reuse tradeoff:** The bash functions being ported are 30-80 lines each — TRIVIAL-REGEX-PORT. Adopting Danger as engine to avoid porting 50 lines of string matching is SSOT #40 Bats reasoning applied in reverse (complexity for minimal gain). The rule: reuse where the external tool's full capability is the match; own-build where only a small slice of external API is needed.

### §5.3 D7 implication (derived from D6=a)

D6=(a) → D7: PR-time surface (`.github/workflows/discipline-self-check.yml`) is **Wave 11 scope**, not Wave 10.

Wave 11 = dangerfile.ts setup (PR-body §1.7 validation using `danger.github.pr.body`) + workflow simplification to `npx danger ci`. This is where Danger's adopted (SSOT #41) value is best realized — PR body is only accessible post-push in CI, exactly Danger's intended domain.

## §7 Fallback spec

### §7.1 Design principle (T-Wave10-A)

Fallback MUST be minimal. TS-core is full. NOT vice-versa. The bash-fallback becoming "rich enough to skip the TS migration" is the primary failure mode.

### §7.2 Critical checks (bash-fallback, ALL non-TS stacks)

Only checks that:
- Are expressible in ≤20 lines of bash
- Cannot silently fail (user must see failure)
- Block the most high-impact violations (Prior-art trailer bypass, §1.7 theatre)

| Check | Fallback implementation | TS-core equivalent |
|-------|------------------------|-------------------|
| Prior-art trailer presence | `grep -l "Prior-art:" <commit body>` — presence only, no substance | `pa_check_trailer()` with substance arm |
| §1.7 trailer presence | `grep -l "§1.7:" <commit body>` — presence only | `s17_check_trailer()` with body-prose arm |
| Node.js availability | `command -v node && [ -f packages/core/hooks/pre-push.ts ]` → dispatch to TS | Feature detection |

### §7.3 Enhanced checks (TS-core only)

| Check | TS-core enhancement | Why not in fallback |
|-------|--------------------|--------------------|
| Prior-art substance arm | SSOT citation validation (not just `skipped —` escape hatch) | Too complex for bash without TS string processing |
| §1.7 substance arm | file:line citation presence | Same |
| §1.7 body-prose detection | `(^|[^/])§1\.7` pattern with URL exclusion | Regex complexity + historical cutoff date comparison |
| Historical cutoff check | Author date comparison vs cutoff | Date arithmetic in bash is fragile |
| actionlint / zizmor | External tool invocation | These require PATH tools; bash bootstrap can call them too but they're not "critical" in the Prior-art sense |
| Principles tests | `npm run test:principles` | Node required |
| Manifest render drift | `npx tsx render/render-rules.ts --check` | Node required |

### §7.4 Feature detection pattern

`.husky/pre-push` 10-line bash bootstrap:
```bash
#!/usr/bin/env bash
REPO_ROOT=$(git rev-parse --show-toplevel)

  && [ -f "$REPO_ROOT/packages/core/hooks/pre-push.ts" ]; then

# Fallback: bash-critical checks only
```

## §9 Adversarial counter-prompt

**Question: "What aspect of the hook architecture did this research miss?"**

**Running the adversarial prompt against the research output:**

**A1 — Windows compatibility:** `.husky/pre-push` uses bash shebang. TS hook via `node --import tsx/esm` works cross-platform. But `packages/core/hooks/pre-push.fallback.sh` bash fallback breaks on Windows. **Mitigation:** fallback is bash by design — Windows users on non-TS stacks are out-of-scope (per project's existing bash dependency in pre-push). The feature-detection dispatch already excludes Windows from `node --import tsx/esm` path if `tsx` is absent, which defaults to bash-fallback. Windows TS stacks (Node.js available) get the TS path and skip bash-fallback entirely.

**A2 — Hook bypass via `--no-verify`:** `git push --no-verify` bypasses all hooks. This is unchanged by TS migration — pre-push hooks are always bypassable. CI gate (`discipline-self-check.yml`) provides the non-bypassable enforcement layer (Wave 11).

**A3 — `tsx` version drift:** `tsx ^4.19.0` is in `packages/core/package.json`. Consumer projects running `install.sh` may not have this version. `install.sh` currently copies hook files — it needs to also ensure tsx is available at the install target. **Gap: install.sh feature detection (D5) must handle tsx version check, not just `command -v node`.**

**A4 — Pre-push hook performance:** Current bash pre-push takes non-trivial time (actionlint + zizmor + npm test:principles). TS bootstrap adds ~200ms cold-start (`node --import tsx/esm` ESM loader). Total performance impact: +200ms per push. Acceptable given target ≤30s hook time stated in pre-push comment header.

**A5 — Stryker on hooks during CI:** `stryker.config.mjs` extended to include `hooks/**/*.ts` — Stryker mutation run is currently manual (not in CI per no-paid-LLM policy). This is correct: Stryker = developer-time tool per SSOT #39. No change to no-paid-LLM policy.

**A6 — `pre-push.test.sh` sourcing pattern:** Current `pre-push.test.sh:24` uses `eval "$(sed -n ... "$HOOK")"` to source functions directly from the bash hook. After migration, the TS functions are imported normally in vitest — no `eval` needed. But during the transition period (Wave 10.1-10.2), the bash hook still exists with the functions. Wave 10.2 must keep bash functions in place until Wave 10.3 deletes them (incremental migration — T-Wave10-B confirmed).

**A7 — `danger local` base branch assumption:** Danger `danger local --base main` assumes `main` is the trunk. If a consumer project uses `master` or a custom branch, `danger local` breaks. Own-build TS hook uses `git rev-list origin/main..HEAD` which has the same assumption but is configurable via env var (`UPSTREAM_REF`). Both options have same fragility; own-build TS is more configurable.

**A8 — What about `packages/core/audit-self/hook-stub-completeness.test.sh`?** This file was NOT in the kickoff §1 table (only 4 files listed). `ls packages/core/audit-self/` shows 5 files including `hook-stub-completeness.test.sh`.
