# Patch 2026-05-08 — Phase 8.8 SSOT format designed without ADR/MADR consult

## Problem

[`prior-art-evaluations.md`](../prior-art-evaluations.md) (introduced as the build-vs-reuse SSOT in Phase 8.8 T1, commit on `docs/phase-8.8-mechanism`) is functionally an Architecture Decision Record register: each entry carries Candidate / Domain / Verdict / Rationale / Trigger to revisit — structurally adjacent to MADR's «Decision drivers / Considered options / Decision outcome» fields and to Michael Nygard's original ADR template. Phase 8.8 designed the format bottom-up from Phase 8 retro lessons; no context7 sweep against ADR / MADR / Y-statements / decision-records / architecture-as-code prior art was performed before T1 shipped the format spec.

The functional outcome (markdown table SSOT, 5 entries shipped, observed-zero-FP through Phase 9 entry research) is unchanged by this finding — the format works, and ADR-adjacent format convergence is a *good* sign, not a problem. But the *recording-side* gap is real: no provenance trail to the precedent body of work, and no audit of the MADR template's specific field choices (e.g. «Negative consequences», «Pros / Cons of the options») that might have improved the SSOT shape.

This is also a **recursive self-application gap**: the discipline being established in Phase 8.8 (consult prior art before building) was not applied to the design of Phase 8.8 itself. The retro's [§4 retroactive audit](../retros/phase-8.8.md) audited Phase 8 against Phase 8.8's SSOT but did not audit Phase 8.8's own design choices.

## Root Cause

- **§1.2 Category sweep — `#category-sweep-missed`.** «Decision records / ADR templates / architecture-as-code» is a well-known category (Michael Nygard 2011, MADR, Y-statements per Olaf Zimmermann, Pierre Pureur's «Continuous Architecture»). The category was not enumerated when designing the SSOT format. Search anchored on the *content* domain (build-vs-reuse for our specific capabilities) rather than the *format* category (decision-record patterns).
- **`#scope-not-formal-trigger`.** Phase 8.8 was an *implementation* phase, not a Step 0 *entry research* phase, so [EXECUTION-PLAN.md §5.5 Step 1.5](../EXECUTION-PLAN.md) did not formally trigger. The §5.5 gate only fires for entry research; the SSOT-format design moment slipped through because no rule mandates prior-art consult on *implementation-phase format decisions*.
- **`#recursive-self-application-gap`.** The Phase 8.8 retro § «Self-application at-creation evidence» cites the principle's recursive self-application (PR #9 ultraview, T2-T11 trailers, principle 08 validating SSOT), but it does not apply the principle to its own *format choice*.

## Solution

This patch records the gap. No retroactive edit to [`prior-art-evaluations.md`](../prior-art-evaluations.md) — the format has shipped and is observably working. If a future session promotes ADR/MADR to a 6th SSOT entry («ADR Decision Records, ADOPT VOCABULARY» or similar, parallel to fitness-functions entry #3), the SSOT itself can carry the precedent. Decision deferred — adding a SSOT entry now would inflate per [§3 of `prior-art-evaluations.md`](../prior-art-evaluations.md) «entry without non-trivial trigger condition is a sign of weak rationale».

The Phase 8.8 retro is merged history per Hard Constraint #10; the gap is recorded here, not edited into the retro.

## Prevention

Operationalisable rules that, if applied at Phase 8.8 T1 time, would have caught the gap:

1. **Format-category sweep on SSOT/register/log/contract/template artifacts.** When introducing a new format spec (markdown table, JSON schema, YAML contract, file-naming convention), enumerate the named precedent categories before designing bottom-up. Concrete: «is this a decision record? a CHANGELOG variant? an ADR? a CODEOWNERS-shape register? a fitness-function table?» — at least one named-precedent search per category.
2. **Implementation-phase format-decision gate.** Extend [§5.5 Step 1.5](../EXECUTION-PLAN.md) (currently entry-research-only) to fire on *any* commit that introduces a new shared format under `docs/meta-factory/` or `packages/core/<new-dir>/`. The gate trigger is «new file with table-of-rows / record-of-decisions semantics», detectable via path glob + content heuristic.
3. **Recursive self-application audit at retro time.** Phase retro Self-reflection block must include one prompt: «Did the principle established in this phase apply to the design choices of this phase itself? If not, what slipped through?» — separate from the existing «what did we miss» self-reflection. Calibrates whether discipline is operating bottom-up (corrective) or top-down (preventive).
