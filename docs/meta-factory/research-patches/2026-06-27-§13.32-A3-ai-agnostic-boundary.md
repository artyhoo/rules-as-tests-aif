## §1 Stream scope

A3 audits the **AI-agnostic boundary** — the claim, made repeatedly across the repo, that the framework's artefacts work when consumed by AI harnesses *other than Claude Code* (Cursor, Codex, Aider, Cline). Three audit dimensions per kickoff §3 A3:

- **(a)** Does the «AI-agnostic» framing hold under inspection, or is an artefact CC-only by a hidden assumption (CC-specific tool names, hook events, slash-command syntax, `.claude/` paths)? — proven with `file:line` evidence.
- **(b)** Is the three-layer authority model (framework / preset / consumer) coherent with current `install.sh` logic? — traced through the actual copy steps.
- **(c)** What external prior art exists on AI-agnostic prompt portability / cross-agent prompt formats as of 2026? — DeepWiki + WebSearch.

**Load-bearing prior work consulted (T3, T13 — do not trust «done» without verifying upstream evidence):** an entire `agnosticism-remediation` umbrella already audited this exact surface and shipped a CI gate. `cat .claude/orchestrator-prompts/agnosticism-remediation/done.md` → `Final PR: #577, Closed: 2026-06-16` (T-A #570 doc-claims probe; T-B #575 portable AGENTS.md rule index; T-C #577 principle 21 conformance test). ## §4 Per-class audit findings

### 4.1 Audit dimension (a) — does «AI-agnostic» hold?

**Reporting-only agents (items 1-6): PASS — genuinely portable.** All six carry only universal tool primitives (`Read / Glob / Grep / Bash / Write`) — present in every harness. Each opens with an explicit portability header: e.g. `compliance-verifier.md:13` — «You are reading this prompt in your **active AI session** (Claude Code, Cursor, Codex, Aider, or any other IDE-integrated assistant)». The bodies reference repo-relative paths (`.claude/rules/...`, `gh pr view`) that any harness with file + shell access can resolve. **No hidden CC-only assumption found in the reporting-only six.**

**The two liveness probers (items 7-8): PARTIAL — portable prose, CC-coupled mechanism + frontmatter.** This is the central A3 finding. Three CC-specific seams:

1. **`tools: Read, Glob, Grep, Agent` frontmatter** (`manual-rule-liveness-prober.md:5`, `shipped-agent-liveness-prober.md:5`). The `Agent` tool and the `tools:` field are **Claude-Code-specific** — DeepWiki on `anthropics/claude-code` (§6 C2): «The `tools:` frontmatter field and the Agent/Task sub-agent dispatch mechanism are specific to Claude Code … deeply integrated into Claude Code's internal systems.»
2. **The core methodology IS sub-agent dispatch.** `manual-rule-liveness-prober.md:75`: «Dispatch a **FRESH subagent** via your harness's fresh-subagent mechanism (Claude Code's Task/Agent tool, Cursor's equivalent sub-agent dispatch, Aider's spawned session, etc.)». The prose *acknowledges* harness-specificity, but Aider has no nested sub-agent dispatch with `tool_uses` introspection — so the RED→GREEN delta the prober computes is not reproducible in every named harness.
3. **`manual-rule-liveness-prober.md:128`: «Must be invoked via top-level `claude --agent`, not as a dispatched subagent».** Hard CC-CLI coupling stated in the artefact itself.

**Counter-prompt (T7, per kickoff §5.4 — «name the concrete CC-only assumption that would break this in Cursor/Aider»):**
> *Prompt:* "Name the concrete CC-only assumption that would break the liveness probers in Aider."
> *Answered with evidence:* The probers require a **nested fresh-subagent dispatch that returns the child's full output + tool-use record** to the parent for delta computation. Aider's architecture (single-session edit loop, per §6 WebSearch) has no such primitive — `tools: Agent` (line 5) is unresolvable, and there is no Aider construct that returns a child agent's `tool_uses` array for the `shipped-agent-liveness-prober` Pass-1 fabrication check (`shipped-agent-liveness-prober.md:107`). The artefact would not *break loudly* (it's a markdown prompt, not executable) — it would **silently under-deliver**: an Aider session reading it cannot perform Step 3-4 and has no degradation path written for that case.

**Mitigating evidence (T16 — verify the actual mechanism, don't pattern-match on the `Agent` tool name):** the seam is narrower than «CC-only». (i) Both probers are **authoring-only, never shipped** — `setup.d/20-agents.sh:25-28` skips them in the default install (`case … manual-rule-liveness-prober.md) continue ;;`) and `install.sh:226-227` skips them on `--refresh`. So the consumer-facing AI-agnostic surface is the **six reporting-only agents**, which pass cleanly. (ii) Codex CLI *does* have an equivalent: DeepWiki on `openai/codex` (§6 C3) confirms an `import_subagents` path that converts `.claude/agents/*.md` to `.toml` plus a native `spawn_agent`/`wait_agent` dispatch. So the prober mechanism is portable to **Codex** (via import) though not to **Aider**. The prose «Cursor's equivalent / Aider's spawned session» over-claims for Aider specifically.

**Verdict (a):** the consumer-shipped AI-agnostic surface **holds**. The two operator-only probers are **CC-favoured with a documented-but-incomplete portability claim** — the gap is Aider-class harnesses lacking nested dispatch, and the `tools: Agent` frontmatter is silently CC-specific.

### 4.2 Audit dimension (b) — three-layer model coherence with install.sh

Traced through the actual copy logic:

- **Layer 1 (framework default):** `$PKG_ROOT/agents/*.md`, `packages/core/templates/shared/...`. Read-only for consumers (INSTALL-FOR-AI.md:243).
- **Layer 2 (consumer in-place edit):** `copy_safe` skips existing files without `--force` (`setup.d/20-agents.sh:29`), preserving edits — matches INSTALL-FOR-AI.md:244.
- **Layer 3 (`.override.md` escape hatch):** install honours it for skills (`install.sh:237-245`, `_override="${_dst}.override.md"`). Codex precedent cited (INSTALL-FOR-AI.md:268, entry #15).

**COHERENT — with one drift to surface (audit b finding):** the model is *internally* coherent and the `--refresh` path (`install.sh:308-316`) correctly documents that Layer-2 in-place edits are NOT preserved by `--refresh` (it is stateless). But there is a **CC-path assumption baked into Layer 1→2 for agents specifically**: all agents copy to **`.claude/agents/`** (`setup.d/20-agents.sh:23`, `install.sh:229`). For a Cursor/Aider/Codex consumer, `.claude/agents/` is not the harness's native agent location (Codex expects import to `.toml`; Cursor uses `.cursor/`). The three-layer *authority* model is harness-agnostic in prose, but the *destination path* of the agent layer is CC-shaped. INSTALL-FOR-AI.md:333-337 is **honest** about the parallel hook layer being CC-specific and inert elsewhere — but makes **no equivalent statement for the `.claude/agents/` destination**. That asymmetry is the audit-(b) gap.

### 4.2b ADOPTED-MECHANISM 5-check — item 12 (AGENTS.md rule index)

Kickoff §4 mandates the 5-check integration audit + T16 wording for any ADOPTED-MECHANISM:

> **Upstream problem class:** a single repo-root markdown file that any AI coding agent reads to learn project conventions (a «README for agents»). **Our problem class:** off-CC there is no `.claude/rules/*` auto-load, so a portable index must name every rule for harnesses that don't auto-discover. **Match? evidence:** identical — both are «one portable instruction file, harness-agnostic». The project uses AGENTS.md for *exactly* its designed purpose. T16 PASS — no name-mismatch.

5-check: (1) **problem-class match** — yes (above). (2) **version/spec current** — AGENTS.md is plain Markdown, no schema, no version coupling (§6 C4). (3) **upstream maintained** — Linux Foundation AAIF, 170+ members, 60k+ repos (§6 C1). (4) **our integration matches best-practice** — `rules-autoload.sh` asserts AGENTS.md lists all 15 rules (live §5: PORTABLE); best-practice is «keep it the single source of truth» (§6 C1) — we do. (5) **drop-in alternative** — `.cursorrules`/`CLAUDE.md` exist but are *less* portable; AGENTS.md is the widest-support choice. **ADOPTED-MECHANISM verdict stands; integration is sound.**

### 4.3 no-paid-LLM policy (item 10) — PASS

`no-paid-llm-in-ci.md` is harness-agnostic by construction: it forbids API-billed calls in CI and routes semantic checks to «AI-agnostic sub-agent prompt … consumed by an active AI session running on the operator's or consumer's own Claude Code / Cursor / Codex / Aider subscription» (line 20). The `paid.sh` probe enforces it deterministically (live run §5: PORTABLE). `#subscription-conflation` anti-pattern (line 57) explicitly handles the «consumer may have no CC subscription» case. No CC-only assumption.

- **C3 (DeepWiki `openai/codex`):** Codex CLI **imports** `.claude/agents/*.md` (`import_subagents` → `.toml`) and has native sub-agent dispatch (`spawn_agent`/`wait_agent`). So sub-agent prompts are portable to Codex **via translation**, not native read.

**Synthesis (the load-bearing external evidence):** the project's **portable doc-index** approach (`AGENTS.md` listing all rules — `rules-autoload.sh` PORTABLE) is **externally validated** — it IS the 2026 industry-standard portability mechanism. The project's «works in Cursor/Codex/Aider» prose on the dispatch-based probers is **partly accurate (Codex) and partly aspirational (Aider)**.

**DECISION-NEEDED A3-1 — `tools: Agent` frontmatter is silently CC-specific on the two probers.**

- *Option B* → leave as-is; the 2 probers are authoring-only and never shipped, so the consumer surface is already clean — the gap is cosmetic.
- Both legitimate; maintainer decides whether «green gate on an unprobed surface» is acceptable theatre-risk.

**DECISION-NEEDED A3-2 — prober prose over-claims Aider portability.**
`manual-rule-liveness-prober.md:75` / `shipped-agent-liveness-prober.md:107` say «Cursor's equivalent sub-agent dispatch, Aider's spawned session, etc.» — but Aider has no nested-dispatch-with-tool_uses primitive (§6 C2/WebSearch). Codex *does* (import path, C3).
- *Option A* → narrow the prose to «Claude Code (native) or Codex (via `import_subagents`); harnesses without nested sub-agent dispatch cannot run this probe — structural gate (principle 02/21) is the fallback».
- *Option B* → keep the inclusive prose as aspirational; accept it as a known soft-claim.

**DECISION-NEEDED A3-3 — `.claude/agents/` destination path has no honest off-CC degradation note (unlike the hook layer).**
INSTALL-FOR-AI.md:333-337 honestly states the hook layer is CC-specific and inert elsewhere. No parallel note exists for the agent layer's `.claude/agents/` destination, which Codex/Cursor do not read natively (Codex needs `import_subagents`; C3).

- *Option B* → no change; consumers are assumed CC-or-Codex per README deployment-surface framing (README.md:8).

**Observation (not DECISION-NEEDED) — the positive finding:** the project's portable **AGENTS.md rule index** is externally validated as the 2026 cross-agent standard (§6 C1/C4). The agnosticism-remediation umbrella (#577) landed the *right* mechanism.
