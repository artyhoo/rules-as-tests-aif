## §1 — Problem (the gap this surveys)

**Origin (kickoff §0):** Session 2026-05-22, the agent repeatedly stumbled over the codified automerge→staging flow *despite it being in memory* ([[project_automerge_staging_plan]]). Maintainer's root insight: **«remember reliably» ≠ «put in memory»**. Memory is stage-0 (unreliable). Reliable = force-fed every prompt **or** mechanically gated — but always-on injection of every rule bloats context. The open design problem: **deliver the RIGHT rule at the RIGHT moment, reliably, without context bloat.**

Before codifying our own channel-selection principle (BFR-default), this patch surveys prior art and adversarially tests the draft principle from kickoff §1:

> **Draft:** Persist each rule at the NARROWEST channel that still reliably catches its violation. Channel breadth ∝ how often the rule is relevant. Ladder: mechanical gate (fires on the action, 0 standing cost) > context-triggered injection (scoped to path/tool/task) > always-on prompt digest (reserve for 3–4 sweeping invariants) > memory (last resort, stage-0).

This is the consumer-facing analog of README's existing **"earliest reachable channel"** invariant — that one orders enforcement in *time* (edit → pre-commit → pre-push → CI → audit); this draft orders rule *delivery* in *scope/cost*.

## §3 — The channel-selection principle: refuted-and-refined

The draft **survives** as direction (every surveyed tool confirms scope/cost-ordered delivery + reserve always-on for sweeping invariants) but its **single linear ladder is wrong on two counts**. Adversarial counter-prompt (T7) run against the draft:

**Refinement A — two orthogonal axes, not one ladder.** The draft conflates *enforcement type* with *scope breadth*. They are independent:
  1. **Detectability axis → choose TYPE.** Is the violation mechanically detectable? **Yes → GATE** (PreToolUse block / pre-push / CI-last-resort): blocks the action. **No (needs judgment) → INJECTION**: delivers the rule's text to inform the decision. A gate and an injection are not rankable as "better/worse" — a judgment rule *cannot* be gated, so injection is not a "weaker gate", it is the only tool. (aif-handoff git-drift = gate; a style convention = injection.)
  2. **Relevance-scope axis → choose BREADTH** of whichever type: narrowest trigger that still **reliably** fires.

**Refinement B (load-bearing) — semantic triggering ranks BELOW always-on on reliability, not above.** The draft puts "context-triggered injection" strictly above "always-on digest". True **only when the trigger is deterministic** (a hook matcher: tool/path/glob — OhMyOpencode `rulesInjector`, Cursor Auto-Attach, CC PostToolUse). When the trigger is **semantic** (Superpowers CSO, Cursor "Agent" type, skills `when_to_use`), it is *best-effort* — the model may not load it. Superpowers' 1% rule is a prompt-level patch *for this exact unreliability*. **Our origin incident is a semantic/memory under-fire**, so promoting semantic delivery for a load-bearing rule would reproduce the bug. Corrected reliability order:

> **deterministic matcher (hook on tool/path/glob)** ≳ **always-on digest** > **semantic/description trigger** (best-effort) > **memory** (stage-0, unreliable).

(`≳`: a deterministic matcher equals always-on when its trigger fires every turn — e.g. a UserPromptSubmit hook is "always-on by another name" — and strictly dominates when the matcher is narrower than every-turn, buying reliability *and* lower standing cost. Caveat on the semantic rung: a deliberately near-universal description ("apply when editing any file") converges back to always-on reliability — so the rung-below-always-on claim holds for *less-than-universal* semantic scope, which is the intended use; it is not a claim that semantic triggers are *always* unreliable.)

**Refined principle (proposed):**
> For each rule, pick the **type** by detectability (mechanically-detectable violation → *gate*; judgment → *injection*) and the **breadth** by relevance-frequency, choosing the **narrowest trigger that still fires deterministically**. Prefer a deterministic matcher (hook) over an always-on digest; use an always-on digest only for the 3–4 sweeping invariants; use semantic triggering only for non-load-bearing convenience; never rely on memory for a rule that must hold. This is the *delivery-scope* companion to README's *enforcement-time* "earliest reachable channel" invariant.

## §4 — ADOPT/REUSE vs the genuine BUILD gap

- **No engine to build.** CC hooks (SSOT #20, ADOPT) already provide both load-bearing rungs natively (Tier 2 file:line). Agent RuleZ / OhMyOpencode confirm the pattern but importing either = a runtime dependency for a native capability → rejected on maintenance grounds.
- **The genuine gap is discipline, not code** — exactly our project's meta-scope. Today our `.claude/rules/*.md` are delivered either always-on-ish (CC auto-load of the whole `.claude/rules/` dir) or via memory (stage-0). Nothing maps each rule to its *narrowest reliable channel*. The deliverable is the **refined §3 principle codified as a rule**, optionally with one **ADAPTED** thin artifact:
  - **ADAPT (optional, thin):** a CC `PostToolUse` rule-injector adapting OhMyOpencode `rulesInjector` — on Edit/Write, glob-match the touched path against a front-matter `globs:` field in `.claude/rules/*.md`, inject only the matching rule. **Integration cost estimate:** ~1 bash hook (≤60 LOC) + a `globs:` front-matter convention + session-cache to avoid re-inject; ~2–3h. Precedent already in-repo: [check-doc-authority.sh](../../../.claude/hooks/check-doc-authority.sh) path-filters internally — generalising it is the whole job. **Defer to maintainer** (reviewer-discipline §2): the principle ships first; the hook is incremental.

- **Solution (backstop):** This patch — prior-art survey + refined two-axis principle (§3).
- **Prevention:** Before persisting any rule, ask: "is the violation mechanically detectable (→ gate) or judgment (→ injection)?" then "what is the narrowest *deterministic* trigger that still fires?" — never default to memory for a rule that must hold.
