## §4 Per-item audit findings (depth-differentiated)

**Header-accuracy sweep: 37/37 docs read, headers extracted, scope-statement compared to body.** Result summary:

- **35/37 carry an accurate `> **Authoritative for:**` header** that matches the body's actual scope. Spot-verified against bodies for all 5 root/meta docs + all 15 rules + 9 skills + 8 agents.
- **2/37 carry NO header at all** (finding F2).
- **1/37 has a header whose scope is arguably narrower than the body** (finding F3 — CLAUDE.md, borderline).
- **Contradicting-authority (b):** none found that lack a subordination marker (§5 details).
- **Frozen-doc (c):** `PROPOSAL.md` is outside the A6 ls-scope but audited per kickoff §3 A6(c) — **CLEAN** (§4 F4).

### F1 — Canonical list under-covers the A6 ls-scope (MAJOR, coverage gap)

`REQUIRED_HEADER_DOCS` enforces headers on only **25 of the 37** A6-scope docs; **12 are unenforced** (§2 table). Concretely, 8 of 9 `.claude/skills/*/SKILL.md`, 2 of 15 `.claude/rules/*.md`, and 2 of 8 `agents/*.md` can drop or mis-state their header and principle 09 stays GREEN. This is exactly the `#missing-authority-header` exposure the rule exists to close, present *inside the rule's own home surface*. Evidence: principle 09 GREEN (`npx vitest run …09-doc-authority-hierarchy.test.ts` → 23 passed) WHILE `ai-doc/SKILL.md` and `story/SKILL.md` carry no header (F2) — the test cannot see them because they are not listed.

### F2 — Two skills carry no doc-authority header (MAJOR)

`grep -nE '\*\*(Authoritative for|NOT authoritative for|Class):\*\*'` returned **zero** header lines for:

- `.claude/skills/ai-doc/SKILL.md` — body confirmed (read lines 1-20): frontmatter + `# ai-doc …` then straight into `## Overview`; no blockquote header.
- `.claude/skills/story/SKILL.md` — body confirmed (read lines 1-20): frontmatter + `# /story …` then `## Steps`; no blockquote header.

Both are canonical skill primary docs (`doc-authority-hierarchy.md §2` "Required for: skills/*/SKILL.md"). They escape enforcement only because of F1 (not in `REQUIRED_HEADER_DOCS`). Note 7 of the 9 skills DO carry compliant headers (`aif-doctor`, `dispatcher`, `harvest`, `pipeline`, `self-reflection`, `template-audit`, `tool-bootstrapping`), so this is drift in 2 files, not a category-wide miss.

### F3 — CLAUDE.md header scope narrower than body (MINOR, borderline — DECISION-NEEDED)

Header (line 5) claims authority for: *"AI-tooling conventions, capability-commit gates, build-vs-reuse discipline, Artifact Ownership Contract."* Body section sweep (`grep -nE '^##+ '`) shows the doc ALSO owns:

- `## PR strategy` (§89) — drive-by-PR discipline
- `## Umbrella closure convention` (§104) — `done.md` schema
- `## Operational conventions` (§125): `### Harness gates`, `### Meta-orchestrator self-review obligation`, `### Phase -1 principle-test allowlist probe`, `### Pre-dispatch in-flight probe`

These orchestration/dispatch/harness conventions are not literally enumerated in the four header scope-items. The doc itself flags several as *"interim home"* / *"Codification target is `~/.claude/skills/…` (agent-uncommittable global skill)"* — i.e. they are knowingly parked here. Whether *"AI-tooling conventions"* is broad enough to subsume *"orchestration obligations + harness gates"* is a **judgment call** (reviewer-discipline §1: A6 must not pick strategy). Surfaced as DN-A6-2.

## §6 Negative-existence claims + 10-item checklist verification

**Claim:** *"No external doc-convention provides a per-document `authoritative-for / NOT-authoritative-for` scope header that prevents two docs claiming the same scope; the closest precedent (ADR/MADR/RFC) records decision lifecycle, not doc-scope authority."*

10-item checklist ([phase-research-coverage.md §1.1-1.10](../../../.claude/rules/phase-research-coverage.md)):

- **§1.1 own-stack:** the pattern is project-internal (`.claude/rules/doc-authority-hierarchy.md`); no harness primitive provides it. ✓
- **§1.2 category:** category = "ADR / decision-record / doc-governance conventions". Surveyed (below). ✓
- **§1.3 semantic / §1.4 adversarial — WebSearch ≥3 phrasings:**
  1. *"documentation authority scope header 'authoritative for' per-document convention"* → no convention surfaced (results = legal/compliance "authoritative copy", project-scope docs — different sense).
  2. *"single source of truth … ADR MADR RFC scope declaration drift"* → ADR/MADR/RFC family; ownership tracked via `Owner:` field + `superseded` status + folder grouping; "directories that don't tend to drift" — but **no per-doc scope-authority header**.
  3. *"prevent documentation goal drift … LLM agent pattern-matching authoritative language"* → arxiv:2505.02709 (the exact source the rule cites), arxiv:2603.03258 "Inherited Goal Drift", AAAI goal-drift eval — corroborate the *mechanism* (context pressure → drift), none provide the *doc-header fix*.
- **DeepWiki ≥3 phrasings (T11 primary channel):**
  1. `adr/madr` — *"per-document header declaring scope authoritative-for / NOT?"* → **"MADR does not have a specific per-document header field that explicitly declares what scope a document is authoritative for … relies on convention and human oversight to prevent conflicting claims."** Scope only implicit via subfolders + `status`/`deciders`.
  2. `joelparkerhenderson/architecture-decision-record` — *"convention for per-doc authority scope so operational docs can't redefine project goal?"* → **"does not explicitly define a doc-authority hierarchy header … authority implied by Status + Deciders + immutability/supersession."**
  3. `keepachangelog/keepachangelog` — repo not indexed on DeepWiki (honest blocker; WebSearch covered changelog-convention space — no scope-authority header there either).
- **§1.5 floor-not-ceiling:** 3 WebSearch + 2 live DeepWiki + 1 blocked = floor met; §1.1-1.4 all hold → close permitted.
- **§1.6 trigger sweep:** within A6 scope; no other §13.x trigger fires from this claim.
- **§1.7 self-reflexive:** A6 introduces no rule — this is an audit patch (no §1.7 obligation; recorded for completeness).
- **§1.9 SSOT existence-check:** the cited SSOT precedent is the rule's own §1 (arxiv:2505.02709), confirmed present in `doc-authority-hierarchy.md §1`.

**Verdict:** claim **HOLDS**. ADR/MADR/RFC are the nearest external precedent for *per-doc ownership + supersession*, but the *doc-scope authority-boundary header* is not provided by any surveyed convention → the pattern is correctly **ADAPTED** (vocabulary + drift-mechanism external; scope-header implementation ours). **T16 problem-class statement:** *Upstream (ADR/MADR) problem class = record a single architectural DECISION + its lifecycle. Our problem class = prevent two DOCS silently claiming authority over the same scope (esp. goal-redefinition by operational docs). Match? NO — adjacent (both "ownership"), different object (decision vs doc-scope). Evidence: DeepWiki #1/#2 above, both explicit "no such header."* → ADAPT verdict stands; no false-precedent.
