## Problem

| **Always-on MCP** (e.g. configured in `~/.cursor/mcp.json` permanently) | Every session, every turn | ~500-2000 input tokens for tool definitions per server, depending on # of tools exposed | Cacheable per Anthropic prompt-cache spec; ≤90% off for repeat reads within 5-min TTL |
| **On-demand MCP** (user-toggled per task) | Only when user activates | 0 when not needed; full cost when active | Same as above when active |
| **Always-loaded skill** (always read on session start) | Every session start | ~50-300 input tokens for SKILL.md primary | Cacheable similarly |
| **On-trigger skill** (loaded when keyword matches) | Only when description triggers | 0 when not triggered; primary doc cost when triggered; cold references on lazy load | Per-load cache; effective TTL bounded |

**Rule-of-thumb table for §11:**

| Task pattern | Win mechanism | Reasoning |
|---|---|---|
| Live external service access required (DB query, GitHub PR, Slack post) | **MCP** | Skill cannot make network calls; MCP wraps the service |
| Documentation lookup for known library | **`context7` MCP** (single MCP; covers most libraries) | One MCP subsumes N library-specific skills |
| Codified procedure / methodology / decision tree | **Skill** | Skill body is prose / code-snippets; loads only when needed; no runtime exec required |
| Stack-specific lint/test patterns | **Skill** (with cold references) | No live data needed; static knowledge |
| Per-task throwaway operation | **Skill** | MCP overhead not justified |

**Implication for rule 4** (skills > MCP where no live-access needed): operationalise via two-question gate at proposal — (1) live network/state access required? → MCP; (2) knowledge codifiable in static prose/snippets? → skill. Both yes → skill primary + MCP fallback for live ops. Matches «context7 + on-trigger skill» pattern this project already uses.

## §6 Confirmation UX modes — O6

context7 motivation: no new queries — UX-mode analysis is comparison against well-documented industry primitives (`npm install --yes`, `brew bundle`, `asdf`, IDE marketplace, Claude Code `--allow/--ask/--exclude` already known from CLI man pages). context7 lookup would add no novel candidate beyond these established UX paradigms.

| Mode | Description | Safety | Friction | Power-user fit | Primary failure mode |
|---|---|---|---|---|---|
| **Strict per-item** | Confirm each MCP/skill individually | Highest | Highest | Low — tedious for 5+ recommendations | Approval fatigue → user blanket-yes |
| **Bulk-approve** | Show full list, single Y/n | Medium | Medium | Medium — matches AIF `/aif` | User skim-approves without reading |
| **Trust-mode** | Skip confirmation entirely (config flag) | Lowest | Lowest | High in trusted environments | Silent install of harmful MCP |
| **Tiered** | Strict for new categories, bulk for already-trusted ones (e.g. once user accepted any GitHub MCP, future GitHub MCPs auto-bulk) | Medium-high | Medium-low | High | Definition of «trusted category» drifts |

**Industry parallels:** `npm install --yes` (trust + project-opt-in); `brew bundle` (bulk-approve); `asdf install` (tiered via `.tool-versions`); IDE marketplace «trust this extension» (per-item + publisher-trust); Claude Code `--allow / --ask / --exclude` (tiered by capability class).

**Recommendation for §11:** **bulk-approve (matches AIF baseline) + strict-per-item escalation when candidate fails security-scan or carries auth keys.** Trust-mode only via explicit config flag. Tiered approach defers — «trust-category drift» (what counts as «a GitHub MCP» when 3rd-party reimplementations appear?) needs usage data to define stable categories.

## §7 Recursive bootstrap — O7

context7 motivation: no new queries — recursive-bootstrap reasoning anchors on GCC stage1 / `rustc` self-host precedent (already cited in [README.md `Methodology`](../../../README.md)) + this session's HARD PREREQUISITE evidence (operator had to ensure context7 connected before launch). The recursive structure is intrinsic, not findable in an external tool's docs.

The recursive question: **the very tool needed to research what MCPs exist (context7) is itself a candidate the bootstrapping rule would propose.** This session needed context7 connected at start (HARD PREREQUISITE in research prompt) — i.e. the researcher cannot research without first having the tool that the rule recommends.

**Three start states:**

| State | Definition | What the rule can do |
|---|---|---|
| **Cold start** | No MCP, no skill installed | Rule cannot run rule 2 (no way to «исследуй какие MCP серверы существуют» without context7 or equivalent) |
| **Warm start** | Only `context7` (or equivalent doc-fetcher) installed | Rule runs rules 1-3 against context7's reach; can propose stack-specific MCPs |
| **Hot start** | Full bootstrap pack installed | Rule operates in delta mode: only re-evaluate when stack changes (rule 5) |

**GCC stage1 / `rustc` self-host analogy** (per [README.md `Methodology`](../../../README.md) which already cites this precedent for the project's own recursive self-application thesis):
- GCC stage1: minimal C compiler bootstraps stage2; stage2 builds stage3 with full features
- `rustc`: precompiled binary builds source-`rustc`; source-`rustc` builds itself
- Tool bootstrapping: minimal `context7` MCP bootstraps research → research surfaces full MCP set → user installs → hot start achieved

**Three implementation paths for review session §11:**

1. **Zero-default.** `setup.sh` installs nothing; rule 2 fails on cold start; user bootstraps manually. Lowest footprint, highest first-run friction.
2. **`context7`-only-in-`setup.sh`** (recommended). Single dep, warm-start default; matches GCC stage1 minimal-bootstrap precedent; no cold-start failure mode for the rule.
3. **Full bootstrap pack.** context7 + GitHub + filesystem + stack-specific MCPs. Highest first-run readiness; risk of installed-but-unused MCPs contradicting rule 4 token-economy.

**Strong recommendation: path 2.** Framework already cites GCC stage1 in its self-application thesis; making bootstrap MCP install the «stage1 of tool bootstrapping» is recursively coherent. Path 3 violates rule 4.
