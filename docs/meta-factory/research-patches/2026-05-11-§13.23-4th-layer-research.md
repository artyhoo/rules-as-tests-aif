## §0 — Build-vs-reuse (O0)

**Verdict: NO end-to-end match. STOP signal NOT triggered.**

Context7 phrasings used (verbatim — for 7.6.b re-run verification):

### Phrasing 1 — `custom rule trailer footer enforcement pre-push hook validate trailer presence`

Library resolved: `/conventional-changelog/commitlint` (Source Reputation: High; Benchmark: 83.88).

**Docs surfaced:**
- Custom plugin rule pattern: `signed-off: ({ raw }) => /^Signed-off-by:/m.test(raw)` — validates presence of RFC822-style trailer via regex on raw commit body.
- Built-in `signed-off-by` rule: presence check for exact trailer text match.
- Husky integration: `echo "npx --no -- commitlint --edit $1" > .husky/commit-msg` — fires on `commit-msg` hook, not `pre-push`.

**Verdict candidate: ADOPT VOCABULARY.** The custom plugin pattern maps directly to a `§1.7:` trailer check. Critical semantic gap: commitlint fires at **commit time** (per-commit, `commit-msg` hook), while our proposed check fires at **push time** (`pre-push` hook, range `origin/main..HEAD`). Push-time enforcement validates N commits in one pass; commit-time would require trailer on every commit unconditionally. The push-time scope predicate is exactly the design challenge §13.23 targets. commitlint solves «enforce trailer on every commit»; we need «enforce trailer only on discipline-introducing commits in push range». **No dep needed** — the grep pattern is reusable in bash.

### Phrasing 2 — `commit parser trailer footer custom plugin hook conventional commits trailer extraction`

Library resolved: `/semantic-release/semantic-release` (Source Reputation: High; Benchmark: 86.83).

**Docs surfaced:**
- Footer conventions for `BREAKING CHANGE:`, `fix:`, `feat:` — informational parsing for version-bump computation.
- Commit message footer separated by blank line from body; `BREAKING CHANGE:` must appear in footer.
- No enforcement gate — semantic-release **reads** trailers to determine version bump; it does NOT block pushes/commits lacking trailers.

**Verdict candidate: DEFER.** Release automation, not an enforcement gate. semantic-release is downstream of commits — it consumes trailer data but has no mechanism to reject commits that lack required trailers at write time. Different problem space (version automation vs. convention enforcement).

### Phrasing 3 — `custom commit hook schema trailer footer enforcement rule validation`

Library resolved: `/commitizen-tools/commitizen` (Source Reputation: Medium; Benchmark: 96.4; Python-based cz-cli).

**Docs surfaced:**
- `cz check --rev-range master..HEAD` — validates all commits in a git revision range against configured schema. Pattern mirrors pre-push range check semantics exactly.
- Custom `schema_pattern` regex for full message structure validation.
- `validate_commit_message()` override in Python class for custom logic.
- Python dependency (`pip install commitizen`) — not Node-native.

**Verdict candidate: DEFER.** The `cz check --rev-range` pattern is the closest behavioral analog to our push-range validation need — this is a vocabulary signal worth citing. However: (a) Python dep is incompatible with our Node-native stack; (b) schema validation targets commit message FORMAT, not trailer PRESENCE conditioned on file-diff content; (c) interactive commit prompts are out of scope. No adoption.

### Phrasing 4 — `git interpret-trailers RFC822 parse extract key value order-independent`

Library: native git builtin (no context7 library ID — git-scm.com documentation). Verified separately.

**Docs surfaced:**
- `git interpret-trailers --parse` outputs structured key:value trailer list.
- **Order-independent** by design: parses by key, not line position.
- `git log --format=%(trailers:key=Signed-off-by)` for per-key extraction.

**Critical finding:** reading the existing `.husky/pre-push` hook (lines 154–201, `pa_check_trailer()` function), the **current implementation uses bash `case "$line" in Prior-art:*)`** — NOT `git interpret-trailers`. This means: (a) the existing parser is line-anchored and order-sensitive; (b) introducing `git interpret-trailers` for the new §1.7 check while retaining grep-based parsing for `Prior-art:` would create a two-parser inconsistency. **Vocabulary adoption** — the RFC822 standard informs trailer format design; no runtime adoption.

**Verdict candidate: ADOPT VOCABULARY.**

### Phrasing 5 — `husky commit-msg trailer validation bash script signed-off-by enforcement`

Library resolved: `/conventional-changelog/commitlint` (same as Phrasing 1 — no new library). Additional signal: raw bash `commit-msg` hook pattern `grep -qE '^Signed-off-by:'` surfaced as standard alternative to commitlint dep.

**Verdict candidate: N/A (same library; confirms grep-based pattern is standard practice).**

## §1 — O1 Scope predicate

**Problem:** which commits «introduce or extend a rule»? Not every commit touching a rule file is discipline-introducing.

### C1 — File-glob match only

Scope: any commit touching `.claude/rules/*.md`, `packages/core/principles/*.test.ts`, or `.claude/skills/*/SKILL.md`.

| Factor | Assessment |
|---|---|
| Pros | Simple to implement; fully deterministic; no diff-content parsing. |
| Cons | HIGH false-positive rate: typo fixes, cross-reference updates, snapshot regenerations, content-only edits all trigger. |
| FP rate estimate | HIGH — every rule-file edit, regardless of whether a rule was introduced, fires. |
| FN rate estimate | MEDIUM — discipline changes in files outside the glob (e.g. `CLAUDE.md`) are missed. |
| Implementation cost | LOW — 3 glob patterns + `git diff-tree --name-only` check. |
| **Verdict** | REJECT as standalone predicate. Use as outer condition in C4 hybrid only. |

### C2 — `## §`-section diff content match

Scope: commit diff (scoped to discipline-file globs) contains a line matching `^+## §` (new section heading added).

| Factor | Assessment |
|---|---|
| Pros | Targets the act of *introducing a new section* — correlates strongly with rule introduction. More precise than C1 alone. |
| Cons | Section renames produce both `^-## §Old` and `^+## §New` — pure renames trigger (debatable FP: renames are often discipline-adjacent). Content-only edits within existing `## §` sections don't trigger (FN). |
| FP rate estimate | LOW-MEDIUM — typo fixes and content edits don't add `## §` headings; only section additions and renames trigger. |
| FN rate estimate | LOW-MEDIUM — extending an existing section via new checklist bullets (no new `## §`) is missed. |
| Implementation cost | MEDIUM — `git show "$sha" \| grep -qE '^\+## §'` on diff output. |
| **Verdict** | ADOPT as inner condition in C4 hybrid. |

### C3 — Commit-message keyword match

Scope: commit subject or body contains `§1.7`, `discipline`, `principle`, `rule`, or `introduce`.

| Factor | Assessment |
|---|---|
| Pros | Zero code-diff analysis; purely message-based. |
| Cons | Relies on author discipline to label commits correctly; HIGH FN (easy to forget). HIGH FP: «fixes rule file formatting», «cites principle 09», «adds Prior-art: rule» all trigger. |
| FP rate estimate | HIGH — many commits mention rule-related terms without introducing rules. |
| FN rate estimate | HIGH — silent rule introductions without keywords slip through. |
| Implementation cost | LOW — one `grep` on commit body. |
| **Verdict** | REJECT as primary predicate. Consider as optional third-level signal in a future v2 predicate; not recommended for v1. |

### C4 — Hybrid: file-glob ∩ `^+## §` section-diff match (RECOMMENDED)

Scope: commit (a) touches a discipline-file glob AND (b) diff contains `^+## §` (new section heading line added).

| Factor | Assessment |
|---|---|
| Pros | Combines specificity (only discipline-file commits) with precision (only commits adding new sections). Typo fixes, cross-ref updates, content-only edits within existing sections — all pass without triggering. |
| Cons | Misses discipline extensions that don't add a `## §` heading (e.g. adding a new rule to a `### Checklist` bullet without a new section). Section renames still trigger (acceptable: renaming a rule section is discipline-adjacent). |
| FP rate estimate | LOW — typo fixes and content edits don't add `## §`; structural refactors of rule files are uncommon. |
| FN rate estimate | LOW-MEDIUM — subsection-only extensions missed; acceptable for v1. |
| Implementation cost | MEDIUM — two bash conditions: `git diff-tree --name-only` glob check + `git show "$sha" \| grep -qE '^\+## §'`. |
| **Verdict** | **RECOMMENDED for v1.** FP rate manageable; FN rate acceptable. |

### C5 — Explicit author-marker trailer `Discipline-introducing: yes`

Scope: commit carries opt-in trailer signalling intent.

| Factor | Assessment |
|---|---|
| Pros | Zero FP — fires only when author opts in. Immune to file-glob ambiguity. |
| Cons | HIGH FN — easy to forget; no mechanical enforcement of the trailer itself (circular). Discipline-theatre: becomes a ritual, not a gate. |
| FP rate estimate | ~0. |
| FN rate estimate | HIGH — defeats the automation purpose. |
| Implementation cost | MINIMAL. |
| **Verdict** | REJECT as primary predicate. Consider as opt-in escalation supplement: if present, trigger check regardless of C4 outcome. |

**O1 Final recommendation: C4 (file-glob ∩ `^+## §` diff match).** The adversarial test against the last-50-commits on `main` is the required load-bearing verification — this research cannot run it (implementation constraint). **7.6.b reviewer or 7.6.c implementer MUST run this test** and confirm FP rate <5% before promoting the hook to blocking (`exit 1`) mode. If FP rate >5% → F2 fallback applies (warn-only permanent per review §8).

## §2 — O2 Bootstrap chicken-and-egg

**Problem:** the commit that adds the §1.7 pre-push hook is within `origin/main..HEAD` push range. If that commit touches discipline files AND adds a `## §` section heading (it will — the hook introduction adds a comment block with `# ── 8. §1.7 trailer check ──` style heading), C4 would trigger it. But the §1.7 trailer check does not exist yet.

### B1 — `§1.7 Bootstrap: <reason>` explicit exemption trailer (RECOMMENDED)

Mechanism: hook checks for `§1.7 Bootstrap:` line in commit body; if present, skips §1.7 check for that commit. Parallel to `Prior-art: skipped — <reason>` in `pa_check_trailer()` (lines 169–192 of pre-push hook).

| Factor | Assessment |
|---|---|
| Pros | Structurally detectable via grep/case (same pattern as existing `Prior-art: skipped`). Explicit author intent. Single-use documented exemption. No circular dependency — trailer can be authored before push. |
| Cons | Author must remember to add it to the hook-introduction commit. |
| FP rate | ~0 on legitimate bootstrap use. |
| FN rate | LOW. |
| Implementation cost | LOW — one additional `case` branch: `§1.7 Bootstrap:*) return 0 ;;`. |
| **Verdict** | **RECOMMENDED as primary bootstrap mechanism.** Clean structural analog to existing escape-hatch. |

### B2 — Commit-subject prefix `chore(bootstrap-§1.7):`

Mechanism: hook reads commit subject; if starts with `chore(bootstrap-§1.7):`, skips §1.7 check.

| Factor | Assessment |
|---|---|
| Pros | Visible in `git log --oneline`. Structurally detectable from commit subject. |
| Cons | Awkward naming; `§` in subject is unusual; harder to generalise. Subject is for description, not metadata — trailers are the established exemption pattern in this project. |
| **Verdict** | ACCEPTABLE alternative; B1 preferred (trailers are the project's exemption idiom). |

### B3 — `git rev-list --count <upstream>..HEAD = 1` heuristic (first-commit exemption)

Mechanism: if exactly one commit is in the push range, skip §1.7 check (assume it's the hook-introduction commit).

| Factor | Assessment |
|---|---|
| Pros | No author action required. |
| Cons | FRAGILE — in Wave 7, the hook-introduction commit (7.6.c) is commit N on the branch (after 7.1, 7.2, 7.3, 7.4 work). The heuristic would not fire. |
| FN rate | HIGH — wrong in almost all real-world sequencing. |
| **Verdict** | REJECT. Too fragile for the actual commit sequencing of sub-wave 7.6.c. |

### B4 — Environment variable `SKIP_§1_7_BOOTSTRAP=1` operator escape hatch

Mechanism: `[ -n "${SKIP_S17_BOOTSTRAP:-}" ] && return 0` at start of §1.7 check function. (Note: `§` cannot appear in POSIX shell variable names; use `SKIP_S17_BOOTSTRAP` or `SKIP_RULE_BOOTSTRAP`.)

| Factor | Assessment |
|---|---|
| Pros | Explicit one-shot operator escape; documented in hook comments; mirrors `SKIP_TESTS=1` patterns common in project scripts. |
| Cons | Easy to abuse — author could routinely set env var to bypass. Requires discipline to use only for bootstrap. |
| **Verdict** | ADOPT as documented secondary fallback (F5 path per review §8). If B1 creates edge cases, operator sets env var for that single push. |

**O2 Final recommendation: B1 (explicit `§1.7 Bootstrap: <reason>` trailer) as primary; B4 (env var `SKIP_S17_BOOTSTRAP=1`) as documented fallback.** The 7.6.c hook-introduction commit carries: `§1.7 Bootstrap: first commit introducing §1.7 pre-push check — cannot carry a trailer it has not yet introduced`.

## §3 — O3 Trailer-format coexistence

**Problem:** new §1.7 trailer must coexist with existing `Prior-art:` trailer. **Key finding from code review:** the existing `pa_check_trailer()` function (`.husky/pre-push` lines 154–201) uses bash `case "$line" in Prior-art:*)` — NOT `git interpret-trailers`. This is line-anchored grep-style parsing.

### T1 — Separate stanzas, grep-based `s17_check_trailer()` (RECOMMENDED)

Mechanism: add a new function `s17_check_trailer()` parallel to `pa_check_trailer()`. Each function independently searches for its own trailer key across the entire commit body. Order of trailers in the commit message is irrelevant — both parsers scan full body.

Sketch:
```bash
s17_check_trailer() {
  local sha="$1" body payload
  body=$(git show -s --format=%B "$sha")
  # §1.7 Bootstrap: exemption
  if printf '%s\n' "$body" | LC_ALL=en_US.UTF-8 grep -qE '^§1\.7 Bootstrap:'; then
    return 0
  fi
  # Require §1.7: trailer
  if printf '%s\n' "$body" | LC_ALL=en_US.UTF-8 grep -qE '^§1\.7:'; then
    return 0
  fi
  printf '%s' "no §1.7: trailer"
  return 1
}
```

**UTF-8 note:** `§` is U+00A7 (0xC2 0xA7 in UTF-8). `grep -qE '^§1\.7:'` works on macOS default locale (UTF-8) and Linux with `LANG=en_US.UTF-8`. Prefixing `LC_ALL=en_US.UTF-8 grep -qE` guards against `LC_ALL=C` environments. Implementation detail for 7.6.c, not a design-level blocker.

| Factor | Assessment |
|---|---|
| Pros | Fully consistent with existing implementation pattern (no new dependency). No refactor of `pa_check_trailer`. Order-independent by construction (grep scans whole body). Easy to review alongside existing hook section §7. |
| Cons | `§` character requires locale-aware grep (minor). Two parallel parse functions (slightly more code than a unified parser). |
| **Verdict** | **RECOMMENDED.** Resolves the review-verdicts §2 Decision 1 Problem 3 concern («parser mechanism is an implementation choice, not a fact») — grep-based is the choice, and it is consistent with the existing implementation. |

### T2 — Merged single trailer with semicolon-separated sub-keys

Mechanism: single line `Prior-art: <content>; §1.7: <content>`.

| Factor | Assessment |
|---|---|
| Pros | One line to write. |
| Cons | Breaks existing `pa_check_trailer()` payload length check — the `§1.7:` suffix is counted in the Prior-art payload length. Modifying the existing parser creates coupling risk. Two semantically independent concerns merged into one trailer line. |
| **Verdict** | REJECT. Coupling harms both parsers; semver-incompatible with existing `pa_check_trailer` length validation. |

### T3 — YAML-block multi-line trailer

Mechanism: `§1.7: |` followed by indented multi-line content (git YAML-block trailer format).

| Factor | Assessment |
|---|---|
| Pros | Rich structured content. |
| Cons | Requires `git interpret-trailers` for reliable parsing of multi-line blocks — ambiguous to grep. Incompatible with existing grep-based parser pattern. |
| **Verdict** | DEFER. Over-engineered for a boolean presence check. |

### T4 — Migrate all trailer parsing to `git interpret-trailers`

Mechanism: refactor `pa_check_trailer()` to use `git interpret-trailers --parse`; add s17 check in same mechanism.

| Factor | Assessment |
|---|---|
| Pros | Order-independent by spec. RFC822 compliant. Unified parser surface. |
| Cons | Refactoring working `pa_check_trailer()` carries risk to the existing capability-commit enforcement layer. Non-trivial shell quoting for multi-key output. |
| Implementation cost | MEDIUM-HIGH — refactor existing + new. |
| **Verdict** | DEFER to future iteration if T1 shows limitations. This is the F3 fallback path (per review §8): if Problem 3 requires parser refactor, 7.6.c ships 2 atomic commits — (a) refactor parser to `git interpret-trailers`, (b) add s17 check. |

**O3 Final recommendation: T1 (separate stanzas, grep-based). No refactor of existing parser needed. Problem 3 is tractable at high confidence — the review-verdicts uncertainty (parser mechanism) is resolved by reading the existing hook implementation.**

## §4 — O4 Discipline-theatre mitigation

**Problem:** if the scope predicate (O1) has unacceptable FP rate, the hook either blocks legitimate work or gets routinely bypassed — both outcomes are worse than not having the check.

### D1 — Warn-only mode for first 30 days (RECOMMENDED)

Mechanism: ship hook with `S17_WARN_ONLY=true` flag. In warn-only mode: print warning but do NOT exit 1. After 30-day calibration window, promote to `exit 1` if measured FP rate <5%.

| Factor | Assessment |
|---|---|
| Pros | Allows FP rate measurement in production against real commits. Authors see the signal without being blocked. Promotion criteria explicit and measurable. |
| Cons | Authors may ignore warnings during calibration window. Promotion requires a follow-up commit. |
| Implementation cost | LOW — one flag variable + conditional `exit 0` vs `exit 1`. |
| **Verdict** | **RECOMMENDED.** Aligns with review §8 F2 fallback (warn-only permanent if FP rate >5%). Consistent with Decision matrix «Spec validation (SHAs)» row which uses soft-warn / MAY verdict. |

### D2 — Severity gradient per file-glob category

Mechanism: `.claude/rules/*.md` → hard fail; `packages/core/principles/*.test.ts` → warn; `.claude/skills/*/SKILL.md` → pass.

| Factor | Assessment |
|---|---|
| Pros | Calibrated enforcement per risk level. |
| Cons | More complex implementation; per-glob severity table must be maintained; FP still present within each category. |
| **Verdict** | CONSIDER for v2 if D1 calibration reveals glob-specific FP concentrations. Not recommended for v1. |

### D3 — Allow-list for trusted commit subject prefixes (RECOMMENDED as D1 complement)

Mechanism: commits with subject prefix `docs(research-patches):` or `chore(snapshot-regen):` are whitelisted — §1.7 check skipped before scope predicate runs.

| Factor | Assessment |
|---|---|
| Pros | Clears known-safe commit categories without changing the C4 predicate. Zero FP for research patches and snapshot regenerations. |
| Cons | Allow-list must be maintained; discipline-introducing commits with matching prefix would slip through (unlikely in practice — these prefixes are conventions with clear semantics). |
| Implementation cost | LOW — prepend allow-list check before C4 scope predicate. |
| **Verdict** | **ADOPT as complement to D1.** Immediate allow-list entries: `docs(research-patches):` (research patches are not discipline rules themselves), `chore(snapshot-regen):` (known non-discipline commits), `chore(prior-art-update):` (SSOT append-only commits). |

### D4 — Bypass-counter logging without blocking

Mechanism: log bypass events to `.git/s17-bypass.log` for telemetry; surface count at next session.

| Factor | Assessment |
|---|---|
| Pros | Telemetry without friction. |
| Cons | `--no-verify` bypasses the hook entirely — the log would not be written. Pre-push hooks do not have a reliable bypass-detection mechanism. |
| **Verdict** | REJECT for v1. Bypass telemetry for pre-push hooks requires a separate mechanism that does not exist. |

**O4 Final recommendation: D1 (warn-only first 30 days, `S17_WARN_ONLY=true`) + D3 (allow-list for research-patch, snapshot-regen, prior-art-update commit prefixes). D2 as future v2 refinement.**

## §5 — Synthesis

**Recommended combination: C4 + B1/B4 + T1 + D1+D3.**

- **O1 (scope predicate):** C4 hybrid — file-glob ∩ `^+## §` diff match. Catches discipline-introducing commits with manageable FP rate for typical repo activity patterns. The adversarial test against last-50-commits on `main` is the required load-bearing verification (cannot be done in this research session).
- **O2 (bootstrap):** B1 (`§1.7 Bootstrap: <reason>` trailer) as primary exemption, mirroring `Prior-art: skipped` escape-hatch. B4 (`SKIP_S17_BOOTSTRAP=1` env var) as documented operator fallback for edge cases.

- **O4 (theatre mitigation):** D1 (warn-only 30 days, `S17_WARN_ONLY=true` flag) + D3 (allow-list for three known-safe commit prefix categories).

**Implementation shape:** ~50–70 LOC delta to `.husky/pre-push` — well under the capability-commit new-file threshold (`status=M` modification, not `status=A`). Ships as 7.6.c Commit 1.
