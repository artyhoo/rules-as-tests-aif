## §2 Cross-stream patterns (the synthesis value-add)

Four recurring themes appear across ≥2 streams that no single stream names as systemic. Each verified against the actual patches below (not copied from the dispatch brief).

### §2.1 Pattern P-1 — Version-currency lag («pinned/snapshotted, then never re-verified»)

A recurring posture: a tool/spec/citation is pinned or snapshotted at one moment, then the pin is treated as permanently valid and never re-swept. Verified instances span 4 streams:

- **A4 §4.2 / §8 DN-A4-1:** AIF runtime pinned to `v2.11.0, context7-verified 2026-05-08` (`aif-comparison.md:2`) — **7+ weeks stale at audit, never re-verified via DeepWiki or live install**; all AIF-runtime-coupled rows (#31, #50, INSTALL-FOR-AI flow) rest on it.
- **A5 §4.4 / §4.2 / §4.3:** `typescript ^5.6.0` repo-wide while **6.0.3 is GA** (one major behind, A5-F3); `ts-morph ^24.0.0` while **latest is 28.0.0** (4 majors behind, A5-F1); ESLint `^10.4.0` core vs `^9.0.0` react-spa (intra-monorepo major skew, A5-F2). A5 explicitly names TS-5.6 as «the root of the compounding lag».
- **A1 §7 gap #5 / §8 DN-A1-3:** external-prior-art citations for P6/P7/P11 (lockfile-reproducibility, idempotent-installer, canonical-example-baseline) were **inherited from PROPOSAL App.C, not freshly re-swept in 2026** — flagged INCONCLUSIVE-needs-LLM.
- **Self-referential instance (the scaffold's own lag):** the kickoff scaffold (authored 2026-05-17) carried stale counts — «11 principles / 4 hooks» — that the §11.3 dispatch-refresh had to correct to «30 principle-test files / 14 hooks» (kickoff line 9). The audit's own brief exhibited the lag it audits.

**Why it's a pattern, not 4 coincidences:** every instance is the same shape — a *point-in-time verification* recorded as a *standing fact*, with no re-verification trigger that actually fires. The project has SSOT «Trigger to revisit» fields, but they are pull-based (someone must look); nothing pushes a re-check at the version boundary.

### §2.2 Pattern P-2 — Stale positioning vs live reality (docs describe an earlier project than what ships)

Several load-bearing docs describe an *earlier* version of the project than the one that currently ships:

- **A1 §5 (P4 counter-prompt) / §8 DN-A1-1:** the headline «research-based rule generation» runs in its **v1 deterministic (no-LLM, curated) form** today; L2 Research-Agent + L3 LLM-synthesis are v2-deferred (architecture.md:91/132). Honestly disclosed in per-layer notes, but a fresh reader of PROPOSAL §1.2/§2 over-reads current capability.
- **A1 §5 (P3 counter-prompt) / §8 DN-A1-2:** PROPOSAL §1.4 frames the project as «plug-in для AIF runtime» — a FROZEN 2026-05-08 snapshot. Live reality: **standalone framework** (7 packages, own `runtime-bridge`, own preset family), with AIF one companion among several (BFR §1.1 satellite doctrine, SSOT #86 AIF REJECT-substrate).
- **A4 §4.6 / §8 DN-A4-2:** the REJECT rationale behind #27/#30/#44/#67/#80 — «no autonomous batch use case / standing-infra overhead at interactive scale» — is **measurably weakening**: autonomous aif dispatch is now live (#97/#109/#111/#112 consume the aif runtime; memory `project_aif_autonomous_fully_configured`). The premise the rejections rest on has shifted.

**Synthesis:** P-2 is the *semantic* sibling of P-1's *version* lag — the same «recorded once, reality moved» failure, applied to project self-description rather than tool pins. A1 and A4 independently hit it from opposite ends (architecture self-description vs SSOT rejection rationale).

### §2.3 Pattern P-3 — Discipline blind-spots inside the enforcement layer's own home (the sharpest, most goal-relevant)

This is the recursive-self-application invariant (kickoff invariant #2; README «documents lie; tests don't») being **partially unmet against the project's own enforcement machinery**. Three independent streams found a green check sitting on an un-probed surface:

- **A6 §4 F1+F2 (MAJOR):** `REQUIRED_HEADER_DOCS` (principle 09) enforces headers on only **25 of 37** A6-scope docs; **12 are unenforced**, and 2 of them — `.claude/skills/ai-doc/SKILL.md` and `.claude/skills/story/SKILL.md` — **actually carry no doc-authority header at all, while principle 09 stays GREEN** (`npx vitest run …09… → 23 passed`). The `#missing-authority-header` exposure the rule exists to close is present *inside the rule's own home surface*. **(Independently confirmed by the orchestrator via command before this summary.)**
- **A3 §5 / §6 N2 (FALSIFIED claim):** the agnosticism conformance gate (principle 21) reports **8/8 PORTABLE** while its probe set **has zero rows for the `agents/*.md` `tools:` frontmatter** — the one surface where the genuine CC-coupling lives (`tools: Agent` on the two liveness probers, `manual-rule-liveness-prober.md:5`). The gate is green on a surface that *excludes* the only finding.
- **A2 §7 / §8 DN-A2-3:** of 30 principle-test files, **20 were classed by filename+slot without opening the test body** — «a single principle test could be a tautology (principle 04 territory) and I'd have classed it OWN-BUILD without detecting the tautology». A tautological/no-op test would pass the structural class-check.

**Synthesis (the goal-critical finding):** all three are the same shape — *a deterministic check passes because its coverage set is drawn narrower than the surface it claims to govern.* This is exactly «documents lie; tests don't» failing **against itself**: the tests don't lie about what they check, but the *manifest of what to check* is incomplete, so GREEN means «the listed subset conforms», not «the surface conforms». Per README this is the project's core thesis; per kickoff §2 invariant #2 it is the recursive-self-application gate. P-3 is the strongest «fix-first» candidate (§4) because it undermines the thesis directly, and unlike P-1/P-2 it is fully mechanically closable (extend the manifest + add the missing probe).

### §2.4 Pattern P-4 — SSOT register drift (verdict recorded ≠ reality shipped)

The prior-art SSOT carries verdicts that no longer match what the repo actually does:

- **A5 §4.6 / §8 DN-A5-4 (MODERATE):** SSOT **#41 records ADOPT for Danger JS**, but Danger was **never installed** (`grep -rn danger .github/ package.json` → no match, no `dangerfile.ts`); the §1.7 check was **own-built in TS-core** (`packages/core/hooks/checks/s17.ts`, SSOT #57). The register claims an adoption that didn't happen.
- **A4 §4.3 / §8 DN-A4-3:** rows **#32/#43/#46/#88/#106** describe AIF patterns as «production-grade / industry-validated» — but per A4 §4.1, **AIF self-cites nothing** (3 DeepWiki phrasings converge: «does not explicitly cite external prior art»). The patterns *are* externally mature, but via sources AIF doesn't cite — so «production-grade» as written is false-precedent (cosmetic on VOCABULARY rows, but inaccurate).
- **A2 §8 DN-A2-1:** `.claude/hooks/adopt-orchestrator-prompts.sh` exists but is **registered in no channel** (not in `settings.json`, not on `claude -w` like `worktree-setup.sh`) — a mechanism present in the tree but wired to nothing.

**Synthesis:** P-4 is P-1/P-2's register-level cousin — the SSOT is an append-only ledger, so a verdict written once persists even after the implementation diverges (Danger ADOPT→own-built) or the evidence basis is found absent (AIF «production-grade»). The append-only property that makes the SSOT trustworthy as history also lets verdict-vs-reality drift accumulate silently.

## §4 Recommended post-audit I-phase sequencing (recommendation, maintainer disposes)

**Recommended order — highest-value-first, with rationale:**

1. **DN-M1 first (enforcement-layer coverage gaps).** Strongest fix-first candidate: it directly undermines the project's own thesis (§2.3 — «documents lie; tests don't» failing against itself), is mechanically closable at low-medium effort, and has no upstream dependency. (DN-M2's Danger reconciliation and DN-M5's unwired-hook decision are both single-file register/tree edits.)
3. **DN-M4 (positioning-drift)** — doc-only, but gate it *after* DN-M2/M3 disposition because the «current reality» a positioning note would describe depends on whether the maintainer bumps versions (DN-M3) and reconciles the register (DN-M2). Sequence last among the act-on set because it is the one requiring a real moratorium-style window (kickoff §9 moving-target warning); rushing it alongside the cheap fixes invites the audit-invalidated-by-mid-audit-refactor effect the kickoff warns against.
5. **DN-M6 (inherited-citation re-sweep)** — optional tail; not load-bearing for anything else.
