Реальный sweep артефактов не выполнялся. CI принимал.

Но в `audit-ai-docs.test.sh` не было `test_D3` — негативного теста, проверяющего что при отсутствии фразы D-3 probe ПАДАЕТ. Обнаружено только при внешнем review (PR #37), не автоматически.

**Incident 3 — EXECUTION-PLAN.md goal drift (2026-05-09).** `EXECUTION-PLAN.md §1` переопределил цель проекта как «recursive self-application is north star», перекрыв `README.md#why-this-exists`. Дрейф не пойман несколько месяцев: проверялось наличие фразы, не соответствие смысла.

Детали в [`research-patches/2026-05-11-d3-downstream-docs-completeness.md`](2026-05-11-d3-downstream-docs-completeness.md).

**Synthesis.** Проект строится на тезисе «documents lie; tests don't» (README.md:25-38). Этот тезис применялся к пользовательскому коду. Любое CI-требование проверяемое **синтаксически** (текст непустой, файл существует, строка содержит подстроку) может быть удовлетворено **формально без содержательного выполнения**. ### §1.1 The fix-the-test antipattern applied to CI compliance

#### Analogy table: tests vs CI requirements

| В тестах (классический антипаттерн) | В CI-требованиях этого проекта (та же шкала) |
|---|---|

### AIF `/aif-verify` + `review-sidecar`

From context7 query 1-3: `/aif-verify` audits build/tests/linting/TODOs and plan-vs-code drift. `/aif-rules-check` fires `FAIL` when an «explicit hard rule is **clearly** violated» — «clearly» = LLM judgement, advisory. The `review-sidecar` (shipped in this project at `agents/review-sidecar.md`) runs «as an external reviewer with no memory of how the code was written» and targets tautological tests, mock-only assertions, missing edge cases.

**Paired-negative-test lens:** AIF's validation suite (context7 evidence) documents negative test categories in `TEST-CASES.md` template and validates negative test cases in AIF's own skill format via `validate.sh`. However: this applies to *skill format validation*, not to *CI-compliance claim validation*. There is no AIF surface that specifically checks «when the `### §1.7 Forward-check applied` PR section contains generic text, does it FAIL?». `/aif-verify` runs toolchain; it does not parse PR description prose sections.

**Verdict: PARTIAL OVERLAP, NOT end-to-end match.** `review-sidecar` could be pointed at PR description as a substrate (see §0.2), but currently it targets diff-level tautological tests, not PR-description §1.7 section substance. Continue to O1-O7.

### CodeRabbit pre-merge checks

From context7 query 4-5: CodeRabbit ships `pre_merge_checks.description` with `mode: off/warning/error`. Custom checks via `.coderabbit.yaml` support natural-language instructions like «All breaking changes must be documented in Breaking Change section». Example custom check:

```yaml
custom_checks:
  - name: "Undocumented Breaking Changes"
    mode: "warning"
    instructions: "Pass/fail criteria: All breaking changes to public APIs must be documented…"
```

**Paired-negative-test lens:** CodeRabbit provides a semantic description validation surface that CAN in principle be given an instruction like «§1.7 Forward-check section must cite file:line references, not just say "Checked — compliant"». However: (1) CodeRabbit is a SaaS product — self-hosting not available; (2) no documented paired-negative sanity for the check itself (LLM judge layer only; no deterministic negative arm); (3) false-positive rate undocumented for this specific instruction pattern.

**Verdict: PARTIAL (semantic layer present, but SaaS + no deterministic negative arm + unknown FP rate).** Useful as an LLM-judge layer to complement a deterministic Option N baseline.

### commitlint semantic validation plugins

From context7 query 6: commitlint supports custom plugin rules for structural trailer validation (ticket presence, sign-off, etc.) via regex on `raw` commit body. This is already documented in SSOT #23 (`ADOPT VOCABULARY`). However: commitlint validates *commit message format*, not *PR description content*. There is no commitlint surface that parses PR body §1.7 sections. Semantic content validation beyond «trailer with valid format present» is not in commitlint scope.

**Verdict: NOT a match for Wave 8 scope** (validates commit trailer format, not PR body section substance). SSOT #23 already covers this.

### Stryker mutation testing

From context7 query 7: Stryker mutates source code and measures what fraction of mutants are caught by the test suite. This is precisely the «paired negative test for user code» discipline already practiced in this project. However: Stryker operates on TypeScript/JavaScript source — it cannot be applied to «bash script prose content of a CI workflow check» or «PR description text sections».

**Verdict: RELEVANT PARADIGM, NOT directly applicable to CI-compliance layer.** Stryker IS the academic foundation for Option N's paired-negative-test approach — it operationalizes mutation-testing adequacy for user code. Wave 8 applies the same *principle* to CI compliance surfaces where Stryker cannot reach (CI workflow logic, PR body text, commit trailer substance).

### Adversarial counter-prompt check (§1.4)

Counter-prompt: *«If a tool existed that validated whether a §1.7 Forward-check PR section actually cited file:line evidence of a real sweep, what would it look like? Where would it live?»*

Result: such a tool would need to parse PR body structured text, compare it against the repository state (which files exist at which lines), and verify that the cited content matches the claimed discipline. No production tool found doing this specifically. CodeRabbit custom checks are the closest LLM-layer proxy, but require SaaS dependency and have no deterministic negative arm.

**O0 Verdict: PROCEED.** No end-to-end match found. Partial overlaps: CodeRabbit (SaaS, LLM-layer), Stryker (principle but different surface). Build-vs-reuse: reuse Stryker *paradigm* (apply mutation adequacy to CI surfaces), reuse `review-sidecar` *substrate* for LLM-layer if needed. Option N (deterministic paired-negative extension) has no production analog for the CI-compliance surface.

## §3 Test oracle problem (O1)

**Context7 queries:**
1. `"mutation testing paired negative test sanity arm CI compliance check failure detection"` — `/stryker-mutator/stryker-js`
2. `"mutation score adequacy criterion paired test effectiveness meta-test CI compliance verification"` — `/stryker-mutator/stryker-js`
3. `"test oracle problem mutation testing adequacy"` (adversarial counter-prompt — context7 did not surface standalone «test oracle» paper, see below)

### Formal problem statement

The **test oracle problem** (Barr et al. 2015 «The Oracle Problem in Software Testing: A Survey») asks: how do you know that a test actually tests what it claims to test? A test that always passes regardless of implementation behaviour provides zero information — it is a **degenerate oracle**. This applies symmetrically to CI requirement checks: a §1.7-presence check that passes when the body reads «Checked existing rules — compliant» is a degenerate oracle for the discipline it claims to enforce.

**Mutation-testing adequacy criterion** (Stryker, academic foundation): a test suite is adequate w.r.t. mutation testing if it kills all mutants. Applied recursively to the CI compliance layer: a CI-compliance check is *adequate* if it returns FAIL when the compliance action is merely simulated (the «mutant» = stub text that satisfies form without substance).

**Metamorphic testing (alternative oracle):** for cases where the expected output is hard to specify (e.g., «was a real sweep performed?»), metamorphic relations provide oracle-free verification. Applied here: if the same AI agent is given the §1.7 forward-check task with and without access to the relevant files, the outputs should differ substantively. If they are identical, the presence of file-access was not actually exercised.

### Production-grade solutions in current use

| Paradigm | Production tool | Applied to user code? | Applicable to CI-compliance layer? |
|---|---|---|---|
| Mutation testing | Stryker (project ships `stryker.config.json`) | YES | INDIRECT — Stryker mutates TS/JS source; CI compliance surfaces are bash/YAML/prose |
| Paired negative tests | `audit-ai-docs.test.sh` (project) | YES (D1-D4 probes) | YES — the exact pattern to extend to all CI surfaces |
| Anti-tautology mutation | Principles 02, 04, 08, 09 mutation arms | YES | YES — same principle applied to principle tests; extend to §1.7 CI workflow |
| Two-AI review | `agents/review-sidecar.md` | YES (diff review) | PARTIAL — review-sidecar targets diff, not PR description; would need extension |
| Semantic description check | CodeRabbit custom checks | NO (SaaS only) | PARTIAL — LLM layer only, no deterministic arm |

## §4 Semantic PR review automation (O2)

**Context7 queries:**
1. `"PR description validation semantic analysis substance check CI integration false positive rate"` — `/websites/coderabbit_ai`
2. `"pre-merge checks description validation PR section content substance paired negative test self-hosted"` — `/websites/coderabbit_ai`

### CodeRabbit

**Evidence from context7:** CodeRabbit ships `pre_merge_checks.description.mode: off/warning/error` and `custom_checks` with natural-language `instructions` field. Example: «All breaking changes must be documented in the Breaking Change section of the PR description and in CHANGELOG.md.»

**Applied to Wave 8:** A custom check instructing «§1.7 Forward-check section must reference specific file:line citations, not generic prose» is technically expressible. CodeRabbit PASS/FAIL maps to `mode: error`.

**Paired-negative-test lens:** CodeRabbit's custom check mechanism is an LLM judge — it produces a verdict based on LLM interpretation of the instruction + PR body. There is no deterministic negative arm: a well-crafted generic text that *sounds* like a real sweep could pass the judge. The judge itself cannot be subjected to mutation testing. **Verdict: partial reuse — judge supplements, does not replace, deterministic paired-negative baseline.** Also: SaaS-only (no self-hosting), cost per-PR, FP rate undocumented for §1.7-specific instructions.

### GitHub Copilot PR review summarize

Not queried specifically — GitHub Copilot PR review summarizes the diff, does not validate PR body section substance. Not a match.

### Danger JS / Reviewdog

Danger JS and Reviewdog provide CI hooks for custom rule enforcement on PRs. Danger can access PR body text (`danger.github.pr.body`) and apply custom rules. This is a deterministic layer (no LLM): you can write `fail('§1.7 Forward-check section appears generic')` if a regex matches stub text. Danger JS is closer to Option N's deterministic pattern than to CodeRabbit's LLM layer — it does not provide semantic understanding but can enforce structural constraints.

**Paired-negative-test lens:** A Danger rule can itself have a paired negative test: unit-test the rule function with a stub PR body, assert it returns `fail`. This is a deterministic arm. **Verdict: strong partial match for deterministic component — a Danger rule for PR body section content is essentially Option N applied at the PR-level.** However, adds an npm dep; the existing `discipline-self-check.yml` workflow is already the deployment surface and can do the same work with bash + grep without Danger.

### Proof-of-work as paired-negative in disguise

The ORIGIN section proposes: «require §1.7 sweep to *produce* an artefact (e.g. a checklist file with file:line citations) that mechanically cannot be generated without doing the sweep.»

This is equivalent to the paired-negative-test pattern but expressed from the positive direction: instead of testing «does the check fail when the sweep wasn't done», you make it *impossible to generate the artefact without doing the sweep*. The artefact IS the negative test's proof of substance.

**Applied to §1.7 Forward-check:** Instead of a generic prose section, require a machine-parseable checklist like:
```
Forward-check-evidence:
  R1: no TS files in this diff; N/A
  R2: no new validate calls; N/A  
  ...
  principle-08: prior-art-evaluations.md#38 added in same commit; .husky/pre-push §7 check: prior-art-evaluations.md#38 row present at grep line 245
```

The checklist format requires specific file:line references. A stub that reads «all principles checked — compliant» cannot satisfy the format. This is deterministic enforcement with no LLM cost.

**SLSA / Sigstore / supply-chain attestation:** These address build *provenance* (who built what, when, from what source). The trust model is: signed attestation = tamper-evident proof of process. Applied to §1.7: a commit-signed §1.7-evidence file would provide tamper-evident proof that the sweep produced specific outputs at specific commits. This is more complex than needed for Wave 8; reserved as a Phase 11+ consideration.

**Option N equivalence:** Verifiable-artifact pattern IS Option N's deterministic arm from the positive direction — you enforce the artifact format such that it cannot be trivially satisfied. The proof-of-work framing and the paired-negative-test framing are dual: one says «format must be unsatisfiable without real work»; the other says «test must FAIL when real work is absent». They converge on the same implementation.

- BUT: the sanity test's bypass input is specified deterministically as «text matching pattern X» where X must be narrow enough to reject generic text but wide enough to accept real sweeps.
- The regex «no file:line citation in forward-check section» catches the bypass input above (no `file.ts:N` pattern).

**LLM layer (Option L):**

## §9 Adversarial failure modes (O7)

### Cost explosion
**Risk:** Every PR touching discipline-bearing files pays for LLM judge → CI cost balloons.
**Mitigation:** Option N baseline has zero LLM cost (deterministic bash/workflow). LLM layer is supplementary, triggers only when deterministic check passes but semantic doubt exists. Cache keyed on PR body fingerprint (sha256 of §1.7 section text): re-check only on edit. Per-PR cold cost bounded by CodeRabbit custom check model — ≤$0.05 per invocation at Sonnet pricing for a short PR body.
**Verdict:** Mitigated by Option N baseline being free; LLM layer is opt-in and cacheable.

### False-positive flood
**Risk:** LLM judge marks REVISE on legitimate, well-documented sweeps → reviewer fatigue → judge becomes ignored.
**Mitigation:** (1) Option N deterministic arm runs first; LLM layer only fires if deterministic check passes and prose is flagged as potentially generic. (2) §13.10 entry #4's FP-rate gate (<20% on 10+ PRs) applies to Wave 8's LLM layer before promotion to blocking. (3) Advisory-non-blocking initial deployment.
**Verdict:** Mitigated by advisory mode + FP gate + Option N primary.

### AI judge gameable
**Risk:** AI writes §1.7 section that satisfies the judge's prompt without satisfying the underlying discipline. Counter-recursive vulnerability — same as the gap being closed.
**Mitigation:** Judge prompt requires file:line citations — structural constraint that is cheaply detectable by regex independent of LLM. Deterministic sanity: PR body must match `\w+\.\w+:\d+` pattern in Forward-check section. This creates a structural signal the LLM cannot fake with generic text.
**Still-open:** A sufficiently motivated AI could fabricate plausible-looking file:line references. Long-term mitigation: verify cited file:line actually exists at review time (the D5 inverse-completeness pattern applied to §1.7 citations). Deferred to Wave 8.2+.
**Verdict:** Partially mitigated by structural constraint; full mitigation requires citation-verification probe (future sub-wave).

### Prior-art trailer escape hatch gaming
**Risk:** AI writes «Prior-art: skipped — refactor only» on a capability commit.
**Mitigation:** This is a separate gap from §1.7 gaming. Wave 8 D5 scope decision (§12) covers whether Pre-art trailer substance check is in scope. The paired-negative test for this: a commit with `Prior-art: skipped — refactor only` that IS a capability commit (e.g., adds a new file ≥80 LOC under packages/) must be rejected by §7 of pre-push. Currently `.husky/pre-push §7` only validates *trailer syntax*, not the *truthfulness of the escape hatch claim*. Full substance-check for this would require CI-side capability detection, which re-opens the diff-scanning logic.
**Verdict:** Scoped to D5 decision. Partial mitigation via escape-hatch placeholder rejection already in `pa_check_trailer()`.

### Recursive cascade
**Risk:** Wave 8 mechanism itself fails its own §1.7 check at introduction. Bootstrap chicken-and-egg.
**Mitigation:** §13.23 established `§1.7 Bootstrap:` exemption marker (`.husky/pre-push:271-286`). The Wave 8 implementation commit uses `§1.7 Bootstrap: first commit shipping the mechanism; mechanism cannot yet validate itself — bootstrapped by this commit itself.` (≥20 chars rationale).
**Verdict:** Mitigated by existing bootstrap exemption mechanism (shipped Wave 7).

### Mechanism scope creep
**Risk:** §1.7 is one symptom; Prior-art trailers, doc-authority headers, D-N probes all have same shape. Does Wave 8 ship one mechanism covering all, or focused §1.7-only?
**Decision required:** D5 in §12. Recommendation: tiered rollout — Wave 8.1 covers §1.7 section substance (highest-frequency incident); Wave 8.2 covers Prior-art trailer escape-hatch; Wave 8.3 covers Authoritative-for header scope accuracy; Wave 8.4 covers D-N probe completeness (D5). Each sub-wave is atomic and independently valuable.
**Verdict:** Addressed by D5 tiered scope decision.

## §15 Self-reflection (per phase-research-coverage.md §2)

**1. Когда ошибся — почему?**
During the O0 sweep, first impulse was to dismiss CodeRabbit as «SaaS, not relevant to this project». The adversarial counter-prompt (§1.4) forced a second pass that found the custom_checks mechanism — which is specifically the LLM-judge layer for PR body validation. The cognitive shortcut: mentally categorizing «SaaS tool for commercial teams» as «not applicable» before checking what specific capabilities it ships. The moment: drafting the first CodeRabbit paragraph before running the pre-merge-checks query.

**2. Мог ли пропускать раньше?**
The pattern is systemic in this project's history — §1 catalogs three prior occurrences of the same shortcut (SSOT #31 AIF `/aif` was initially overlooked as «dep not candidate»; Incident 4 `DOWNSTREAM_DOCS` was not grepped because «I remember the docs»). The same lazy-evaluation of «known context» vs «adversarial sweep» recurs. It is not a one-off.

**3. Как не пропускать?**
Map to §1.4 (adversarial counter-prompt on negative-existence claims). Concrete new heuristic: *before closing «Tool X doesn't apply to this surface», write down what the tool WOULD look like if it DID apply, then check that description against the tool's actual docs.* CodeRabbit custom_checks is exactly what a «PR-body validation tool» would look like — checking that first would have found it in the first query.

**4. Какой урок?**
«Before stating `Tool X does not cover surface Y`, describe what coverage would look like, then run one targeted context7 query against that description.» Operationalisable: add to `phase-research-coverage.md §1.4` as a companion to the existing adversarial counter-prompt pattern — specifically for SaaS/platform tool dismissals.

**5. Did the principle apply to its own design choices?**
The proposed Wave 8 mechanism was subjected to §8 recursive self-application audit, which found that the LLM-only path fails its own paired-negative-test criterion — this is exactly the `#discipline-theatre` anti-pattern it is trying to prevent. The research caught this recursion before closing, so the recommendation is adjusted to mandate Option N deterministic baseline before any LLM layer. The self-application gap was detected by the research process (§8), not by a reviewer catching it after the fact. This is the successful bootstrap: §1.7 applied to its own design choices, in real-time, during the research session.
