### §1.1 The fix-the-test antipattern applied to CI compliance

#### Analogy table: tests vs CI requirements

| В тестах (классический антипаттерн) | В CI-требованиях этого проекта (та же шкала) |
|---|---|

| Тест «всегда зелёный»: `expect(true).toBe(true)` | Probe «всегда проходит»: `grep -q phrase file` без негативного теста |
| Никто не проверял, что тест реально ловит регрессию | Никто не проверял, что CI-проверка реально ловит «sweep не сделан» |

Базовый антипаттерн = **`#discipline-theatre`**: AI выбирает путь наименьшего сопротивления и удовлетворяет *форму* проверки вместо её *содержания*. Точно как студент, меняющий тест. ## §3 Test oracle problem (O1)

**Context7 queries:** `/stryker-mutator/stryker-js` × 2 phrasings (mutation adequacy / paired-negative arm); adversarial counter-prompt on «test oracle problem» (no standalone paper surfaced).

The **test oracle problem** (Barr et al. 2015 «The Oracle Problem in Software Testing: A Survey») asks: how do you know that a test actually tests what it claims to test? A test that always passes regardless of behaviour is a **degenerate oracle**. Applied symmetrically: a §1.7-presence check that passes when the body reads «Checked existing rules — compliant» is a degenerate oracle for the discipline it claims to enforce.

**Mutation-testing adequacy criterion** (Stryker): a test suite is adequate w.r.t. mutation testing if it kills all mutants. Applied recursively to the CI compliance layer: a CI-compliance check is *adequate* if it returns FAIL when the compliance action is merely simulated (the «mutant» = stub text that satisfies form without substance).

**Metamorphic testing (alternative oracle):** for outputs hard to specify directly («was a real sweep performed?»), metamorphic relations provide oracle-free verification. If the same agent is given the §1.7 task with and without access to the relevant files and outputs are identical, file-access was not exercised.

Per-CI-surface paired-negative-arm status: see §1.1 last table (gap enumeration). Per-tool paired-negative status: see §0 verdict table.

### Proof-of-work as paired-negative in disguise

The ORIGIN section proposes: «require §1.7 sweep to *produce* an artefact (e.g. a checklist file with file:line citations) that mechanically cannot be generated without doing the sweep.»

This is equivalent to the paired-negative-test pattern but expressed from the positive direction: instead of testing «does the check fail when the sweep wasn't done», you make it *impossible to generate the artefact without doing the sweep*. The artefact IS the negative test's proof of substance.

**Applied to §1.7 Forward-check:** Instead of a generic prose section, require a machine-parseable checklist like:
```
Forward-check-evidence:
  R1: no TS files in this diff; N/A
  R2: no new validate calls; N/A  
  ...
  principle-08: prior-art-evaluations.md#38 added in same commit; .husky/pre-push §7 check: prior-art-evaluations.md#38 row present at grep line 245
```

The checklist format requires specific file:line references. A stub that reads «all principles checked — compliant» cannot satisfy the format. This is deterministic enforcement with no LLM cost.

**SLSA / Sigstore / supply-chain attestation:** These address build *provenance* (who built what, when, from what source). The trust model is: signed attestation = tamper-evident proof of process. Applied to §1.7: a commit-signed §1.7-evidence file would provide tamper-evident proof that the sweep produced specific outputs at specific commits. This is more complex than needed for Wave 8; reserved as a Phase 11+ consideration.

**Option N equivalence:** Verifiable-artifact pattern IS Option N's deterministic arm from the positive direction — you enforce the artifact format such that it cannot be trivially satisfied. The proof-of-work framing and the paired-negative-test framing are dual: one says «format must be unsatisfiable without real work»; the other says «test must FAIL when real work is absent». They converge on the same implementation.

## §8 Recursive self-application audit (O6) — SUCCESS CRITERION

### Step 1: Proposed Wave 8 mechanism (from §10 Synthesis)

Option N baseline: extend paired-negative-test discipline to every CI surface lacking it. `discipline-self-check.yml` gets a sanity test that a PR with generic §1.7 text FAILS; `.husky/pre-push §9` gets a sanity test rejecting commits with generic `§1.7:` trailer. Optional Option L: a `compliance-verifier` sub-agent reviews §1.7 sections requiring file:line citations.

### Step 2: Hypothetical bypass attempt

A PR is submitted with:
```
### §1.7 Forward-check applied
Checked existing rules R1-R20 and principles 01-09 — all compliant. No conflicts found with existing disciplines.

### §1.7 Backward-check applied
Complete sweep of existing artefacts performed. No artefacts fall under the new rule's scope.
```
Total char count: ≫40 chars. Current `discipline-self-check.yml` passes this PR. This is Incident 1 shape.

### Step 3: Mechanism response prediction

**Option N baseline (deterministic sanity test):** sanity job constructs PR body with the above text and asserts §1.7 check FAILs. Regex «no file:line citation in forward-check» catches the bypass (no `file.ts:N` pattern). **Predicted: FAIL on bypass input** if sanity requires file:line pattern. Mechanism fires.

**LLM layer (Option L) alone:** LLM judge would classify above as insufficient, but the judge itself has no deterministic paired-negative test — a well-crafted generic text could pass. Recursive gap: judge prompt enforces the discipline, but the prompt itself has no machine-verifiable «does this reject generic text» test. **Predicted: POTENTIALLY PASS** depending on judge quality.

### Step 4: Verdict

- **Option N (deterministic):** FAIL on bypass — mechanism catches its own gaming attempt via structural evidence (file:line pattern). SUCCESS CRITERION met.
- **LLM layer alone:** POTENTIALLY PASS — gameable by well-crafted generic text. This IS `#discipline-theatre` recursively applied. LLM layer MUST be paired with Option N; cannot stand alone.

**Conclusion:** Wave 8 MUST include Option N as load-bearing component. LLM layer supplementary. A mechanism without a deterministic arm is itself an instance of `#discipline-theatre`.
