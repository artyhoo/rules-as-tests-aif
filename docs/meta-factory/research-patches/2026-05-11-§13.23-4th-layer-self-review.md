## Problem

[`.claude/rules/phase-research-coverage.md §1.7`](../../../.claude/rules/phase-research-coverage.md) introduced §1.7 forward+backward check as a process requirement for discipline-introducing recommendations. Three enforcement layers shipped in branch `process/recommendation-self-discipline` (2026-05-09):

1. Process rule prose (`phase-research-coverage.md §1.7`)
2. Skill auto-trigger ([`.claude/skills/self-reflection/SKILL.md`](../../../.claude/skills/self-reflection/SKILL.md))
3. CI PR-description gate (`.github/workflows/discipline-self-check.yml`)

A 4th layer — pre-push per-commit trailer check — was deferred. Status: **deferred through Wave 6** per [open-questions.md §13.23](../open-questions.md). Wave 7 sub-wave 7.6 closes the deferral.

---

## Root Cause

Pre-Wave-7 enforcement had two documented gaps:

**Gap 1 — `local-push-bypasses-CI`:** An operator can push directly to a branch with `--no-verify` (skipping all hooks) or push commits that bypass CI (e.g., direct-to-branch pushes without a PR). In these cases, the CI PR-description gate (Layer 3) fires only at PR review time, allowing a window where a discipline-bearing artefact lands on the branch without any §1.7 trailer. The gap is not theoretical — direct pushes are standard workflow in this repo for in-progress work.

**Gap 2 — `discipline-theatre` risk:** Over-enforcement creates the inverse failure. If the scope predicate (which commits trigger §1.7 checking) is too broad — firing on every rule-file edit regardless of whether a rule was introduced — the check becomes noise, gets routinely bypassed, and degrades from enforcement to ritual. Per [open-questions.md §13.23 «Why deferred»](../open-questions.md): «per-commit `Forward-check`/`Backward-check` trailer enforcement is a non-trivial design problem unresolved at bootstrap».

Both gaps were addressed by the Wave 7 sub-wave 7.6 research+review cycle (7.6.a + 7.6.b → 7.6.c).

---

## Solution

Pre-push hook section 9 in [`.husky/pre-push`](../../../.husky/pre-push) enforces §1.7 trailer at push-time on commits matching the **C4 scope predicate**:

```
FIRES when ALL of:
  (a) commit touches .claude/rules/*.md OR packages/core/principles/*.test.ts
      OR .claude/skills/*/SKILL.md  [file-glob]
  (b) diff contains ^+## § heading OR ^+export const [A-Z_]+:  [content-marker]
  (c) commit subject NOT in D3 allow-list  [exemption]
      (docs(research-patches): / chore(snapshot-regen): / chore(prior-art-update):)
```

**Gap 1 addressed:** push-time gate fires before remote sees the commit, regardless of whether a PR exists. The only bypass is `--no-verify` (documented bypass, not silent).

**Gap 2 addressed:**
- C4 combines file-glob specificity with content-marker precision — typo fixes, cross-ref updates, and content-only edits within existing sections do NOT fire. Adversarial test: 0/50 false positives on last-50-commits of `main`.
- D1 warn-only default (`S17_WARN_ONLY=true`) for 30-day calibration window (through 2026-06-10). Authors see signal without being blocked; promotion to hard-fail after measured FP rate confirmed <5%.
- B1 bootstrap exemption (`§1.7 Bootstrap: <≥20-char rationale>`) resolves the chicken-and-egg: the commit introducing the §1.7 check itself is exempt, documented explicitly.

---

## Prevention rule

> **Before introducing or extending any rule/principle/discipline that has multi-layer enforcement, explicitly identify the failure modes that each enforcement gap permits.** Use the §13.23 promotion path (`open-questions.md` §13.23 trigger conditions #2 and #3) as the canonical trigger format: trigger fires when «a design proposal for the deferred layer materialises» OR «adjacent hook surface widening makes incremental addition low-risk». Do not wait for a documented bypass incident — the gap is the trigger, not the incident.
