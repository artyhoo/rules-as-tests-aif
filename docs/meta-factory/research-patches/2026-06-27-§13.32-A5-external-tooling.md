### §4.1 Vitest `^4.1.8` / `^4.1.5`

1. **T16:** upstream = «Vite-powered test runner for TS/ESM»; ours = same (principle tests, hook tests, rule tests). **Match: exact.**
2. **Version:** latest **4.1.9** (pub 2026-06-15, [npm](https://www.npmjs.com/package/vitest)); `^4.1.8`/`^4.1.5` caret resolves to 4.1.9. **Current.** (5.0.0-beta exists — not stable.)

5. **Drop-in alt:** Jest (would break Stryker vitest-runner + ESM). Bun test (immature for this AST workload). No reason to switch.

**Verdict: KEEP. No finding.**

### §4.2 ts-morph `^24.0.0` — version lag (4 majors behind)

1. **T16:** upstream = «TS compiler-API wrapper for AST read/edit»; ours = read+analyse (`probes/audit-r4.ts`, `audit-ai-docs.ts`) + write-back (`install/wire-eslint-r2.ts`). **Match: exact** (SSOT #131).
2. **Version:** pinned `^24.0.0`; **latest = 28.0.0** ([npm](https://www.npmjs.com/package/ts-morph), pub ~2 mo ago). **4 majors behind.** v28 carries a breaking change for **TypeScript 6.0**. Caret `^24` will NOT auto-update across the major boundary.
3. **Maintained:** yes — active (`dsherret/ts-morph`, 4279 dependents).
4. **Best-practice:** `Project` + `getExportedDeclarations` + `SourceFile` usage is canonical; no off-label use found.
5. **Drop-in alt:** magicast REJECTED (#134 — SpreadElement gap); native TS compiler API (heavier). ts-morph is the right tool.

**Finding A5-F1 (MINOR–MODERATE):** ts-morph is 4 majors stale. Because the project pins `typescript ^5.6.0` (see §4.4) and ts-morph v25-28 track newer TS, the lag is *internally consistent* but compounds the TypeScript-currency gap. Security exposure is low (dev-only AST tool, no network). **DECISION-NEEDED** (§8).

### §4.3 ESLint version skew — `^10.4.0` (core) vs `^9.0.0` (react-spa)

1. **T16:** upstream = «pluggable JS linter»; ours = host for own R-rules + preset rules. **Match: exact.**
2. **Version:** latest **10.5.0** ([eslint.org blog 2026-04](https://eslint.org/blog/2026/04/eslint-v10.2.0-released/)); core `^10.4.0` → resolves current. **But `preset-react-spa` pins `eslint ^9.0.0`** — a full major behind, and `typescript-eslint ^8.59.0` officially supports `^8.57 || ^9 || ^10` ([typescript-eslint.io](https://typescript-eslint.io/users/dependency-versions/)), so the v9 pin is *supported* but inconsistent with core.
3. **Maintained:** yes (ESLint ships monthly).
4. **Best-practice:** flat-config used throughout; correct for v9+.
5. **Drop-in alt:** Biome / oxlint (do not host custom AST rules with the same maturity). No switch warranted.

**Finding A5-F2 (MINOR):** intra-monorepo ESLint major skew (core 10 / react-spa 9). Not a security issue (both receive patches), but a consistency/foot-gun risk: a rule authored against ESLint 10 AST context API could behave differently under the react-spa v9 host. **DECISION-NEEDED** (§8) — align both to 10, OR document the deliberate split.

### §4.4 TypeScript `^5.6.0` — whole repo one major behind

1. **T16:** exact (compiler).
2. **Version:** pinned `^5.6.0` across **all 6 packages**; **latest stable = 6.0.3** ([npm](https://www.npmjs.com/package/typescript), TS 6.0 = March 2026, transition release ahead of the Go-rewrite TS 7.0). Caret `^5.6` will NOT cross to 6.x. **One major behind.**
3. **Maintained:** yes.
4. **Best-practice:** `tsc --noEmit` typecheck per package — standard.
5. **Drop-in alt:** none (it is *the* compiler).

**Finding A5-F3 (MODERATE):** repo-wide TS pinned to 5.6 while 6.0 is GA. This is the **root of the compounding lag** — ts-morph v28 (§4.2) needs TS 6, typescript-eslint tracks TS 6, but the repo holds 5.6. TS 6.0 is explicitly a migration-prep release for the TS 7.0 Go port; deferring it risks a larger jump later. No acute security exposure (dev-time compiler). **DECISION-NEEDED** (§8) — coordinated TS 5.6→6.0 bump is a non-trivial cross-package change, not a drive-by (T5/T-Phase10-A: surface, do not act).

### §4.5 Husky — ADOPTED-VOCABULARY, not a dependency (false-precedent surface)

The kickoff §3-A5 names «Husky» as a load-bearing tool to audit. **Investigation result: the project does not use the `husky` npm package.**

- `grep '"husky"' package.json packages/*/package.json` → **no match** (no dep).
- Install is `git config core.hooksPath .husky` (Makefile:22) + `chmod +x` (Makefile:21).
- `.husky/` is a **directory name**; `pre-push`, `pre-commit`, `post-checkout` are hand-rolled shell scripts.
- DeepWiki (`typicode/husky`, 2026-06-27) confirms: **«Husky is a wrapper that automates `core.hooksPath` … You can achieve similar results manually by setting `core.hooksPath` yourself.»** → the project did exactly that.

**Cross-check the SSOT:** SSOT mentions «husky» only inside *negative-existence* rationales (#56, #57, #176 — «surveyed lint-staged, husky, Lefthook … none solve our class»). There is **no SSOT entry adopting husky-the-tool.** So the framing is internally honest in the SSOT, but the **kickoff's naming of Husky as a load-bearing dep is inaccurate** — it is ADOPTED-VOCABULARY (the `.husky/` convention) with a fully homegrown bash implementation.

**Finding A5-F4 (INFORMATIONAL / anti-theatre win):** «Husky» as load-bearing tool is a **false-precedent name**. Maintenance benefit: zero dep to track for security/version. No action needed beyond noting that the manual `core.hooksPath` approach forgoes husky's `prepare`-script auto-install (consumers must run `make install-hooks` — already the documented flow). **No version/security finding.**
