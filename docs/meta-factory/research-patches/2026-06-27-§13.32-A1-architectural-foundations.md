## §4 Per-class audit findings (depth differentiated per kickoff §4)

### OWN-BUILD (full audit: external-lit sweep + problem-class + alternative enumeration)

- **P2 §1.3, P9 §7, P10 §8, P12 §10, E1 §1, E3 §5.5, T2t agents/** — these are *internal artefacts* (motivation prose, stability boundary, tier scheme, roadmap, goal pointer, prior-art-consult gate, sub-agent file inventory). For an *architectural-foundations* audit, the load-bearing OWN-BUILD claims needing external-existence checks are: **E3 (the prior-art-consult gate as an architectural mechanism)** and **P9/P10 (invariant-core stability + confidence tiers as design constructs)**. External sweep (§6 NE-3, NE-4): no off-the-shelf «prior-art-consult gate for a rule-generation framework» exists as a packaged tool — it is a thin project convention over SSOT + context7/DeepWiki, correctly OWN-BUILD, low risk. P9 immutability-of-core is a project-specific application of the general «semver-for-core + stability tests» idea (PROPOSAL §7 → core-stability.md). **Finding: OWN-BUILD assignments are honest; none duplicate a production-grade tool.** Coverage 7/7.

### ADAPTED (sweep modifications + verify problem-class match T16)

For each, the **upstream pattern is out-of-scope to re-audit** (kickoff §4); the audit verifies the problem-class match and that PROPOSAL *cited* the precedent:

- **P4 §2 6-layer + P5 Path A/B.** Upstream problem class: indirect-prompt-injection-safe agentic code generation (Plan-then-Execute / PCFI). Our problem class: untrusted-doc-fed rule generation where L2 Research (Planner, no file write) feeds L3 Synthesizer (Executor, file write). **Match: YES** — EXECUTION-PLAN §3.4:120-124 explicitly maps L2=Planner / L3=Executor and cites OWASP/PCFI (EXECUTION-PLAN.md:809 `arxiv:2603.18433`). External lit (§6 NE-2) confirms Plan-then-Execute is the *recommended 2026 defense* — adoption is current, not stale.
- **P8 §6 + P13 §15 self-application.** Upstream: compiler self-host/bootstrap (GCC/rustc) + dogfooding. Our class: a discipline framework validating its own rules as executable tests. **Match: YES, partial** — bootstrap proves *self-compilation*; ours proves *self-enforcement*. README#why-this-exists already flags this as «quality signal, not the goal» (analogy is acknowledged-as-analogy, not over-claimed). Honest.
- **P1 §1.2, P6 §4, P7 §5, P11 §9, E2 phases, T1t packages/, T4t .claude/-vs-docs.** All are mainstream patterns (lockfile-reproducibility, idempotent-installer, monorepo core/preset split, linear pipeline, hot/cold doc split). T1t externally confirmed mainstream (§6 NE-5). No problem-class mismatch found. Coverage 11/11.

### ADOPTED-VOCABULARY (sanity-only: vocab consistent + no false-precedent claim)

- **P3 §1.4 «plug-in для AIF».** Sanity check: does the «plug-in для AIF» claim over-state coupling? DeepWiki on `lee-to/ai-factory` (§6 NE-1) confirms AIF is a **workflow/skills runtime** (`/ai-factory.implement`, `.verify`, `.evolve`) that *does not* generate ESLint rules or do recursive self-validation. The project's vocabulary use («logical self-application layer over AIF workflow self-application») is **consistent** with what AIF actually is — no false-precedent. **Caveat surfaced as DN-A1-2:** PROPOSAL §1.4 frames the project as «plug-in для AIF runtime», but the live repo's BFR satellite doctrine (`.claude/rules/build-first-reuse-default.md §1.1`) now treats AIF as one *companion among several* (REJECT-substrate per SSOT #86 per memory). The §1.4 «plug-in» framing is a FROZEN 2026-05-08 snapshot that current strategy has moved past.
- **T3t skills/ boundary.** CC «skills» vocabulary used verbatim; placement (`.claude/skills/`) is the CC convention. Consistent, no divergence claim. Coverage 2/2.

## §6 Negative-existence claims + 10-item checklist verification

Each NE claim below ran **DeepWiki ≥3 effective probes and/or WebSearch ≥3 phrasings** (kickoff §5.5; context7 supplementary only). Checklist items §1.1 own-stack, §1.2 category, §1.3 semantic, §1.4 adversarial applied; §1.5 floor-not-ceiling honoured (closed only where §1.1-§1.4 held).

**NE-1 — «AIF does not itself do stack-aware rule generation + recursive rule-as-test self-validation» (basis for P3 ADOPTED-VOCABULARY).**
DeepWiki `lee-to/ai-factory`: *«AI Factory does not directly generate ESLint or linting rules adapted to a detected stack, nor does it apply recursive self-application where the framework validates its own rules as executable tests… it is primarily a workflow/skills runtime… `/ai-factory.evolve` modifies SKILL.md files directly… rather than being formally tested against themselves.»* → claim **CONFIRMED**. §1.1 own-stack: AIF is the project's named upstream — checked directly. §1.4 adversarial («maybe evolve IS self-validation?») → answered: evolve edits prose, no executable self-test.

**NE-2 — «Planner-Executor / Plan-then-Execute is established, current 2026 external prior art (basis for P4 ADAPTED, not OWN-BUILD).»**
WebSearch ×3 («LLM Planner-Executor pattern… prompt injection defense indirect», phrasings 2-3 on architectural separation): returned `arxiv:2509.08646` (*Architecting Resilient LLM Agents: Secure Plan-then-Execute*), `arxiv:2603.18433` (PCFI — already cited at EXECUTION-PLAN.md:809), emergentmind «Prompt Injection Defense Pipeline». → P4/P5 are **correctly ADAPTED with cited prior art**, not invented. Existence claim is *positive* (pattern exists) — direction favourable to the audit.

**NE-3 — «No off-the-shelf production tool does stack-aware research-based ESLint-rule generation WITH self-validation of generated rules» (the project's core niche claim, EXECUTION-PLAN §3.3).**
WebSearch ×3 («stack-aware research-based ESLint rule generation with self-validation framework 2026»; «generate linting rules from principles instead of presets adaptive to framework version»; «AI-generated ESLint rule self-validation mutation testing negative test gate»): closest hits are *ESLint Rule Composer* (generates a rule + RuleTester from an AST-Explorer pattern — **no research/stack-detection step, no self-validation gate over the generated corpus**) and Meta's LLM-mutation-testing (validates tests, not generated lint rules). DeepWiki `gptlint/gptlint`: *«gptlint does not generate ESLint rules adapted to the detected stack/framework version… nor apply a self-validation step to generated rules… it uses LLMs to enforce coding best practices based on predefined markdown rules.»* → **niche claim holds**: no surveyed tool combines (detect → research → synthesize → self-validate → install). §1.5: floor-not-ceiling honoured — 3 WebSearch phrasings + 1 DeepWiki, adversarial §1.4 («does gptlint's eval system count as self-validation of generated rules?») answered NO (it evals *predefined* rules). Closing at this depth is permitted because §1.1-§1.4 held.

**NE-4 — «No packaged "prior-art-consult gate" tool exists for rule-generation frameworks (basis for E3 OWN-BUILD).»**
Covered by NE-3 sweep + own-stack check: the gate is a thin project convention (SSOT + context7/DeepWiki/WebSearch consult), not a candidate for a packaged upstream. §1.1 own-stack: Claude Code + DeepWiki/WebSearch already provide the research primitives; the *gate wiring* is correctly OWN-BUILD. Low-confidence-but-low-stakes; flagged INCONCLUSIVE-needs-LLM only if E3 were load-bearing for a recommendation (it is not in A1).

**NE-5 — «Monorepo core/preset split is mainstream, not novel (basis for T1t ADAPTED).»**
WebSearch («monorepo packages core preset split linting framework architecture pattern»): Feature-Sliced-Design monorepo guide + better-auth `@better-auth/core` («foundational primitives shared across all packages, no framework-specific code») confirm the core/preset/shared-tooling split is a **standard 2025-2026 pattern**. → T1t correctly ADAPTED (mainstream), not OWN-BUILD. Favourable direction.
