> **Authoritative for:** the S4 stage design of the generator-forbid-mvp umbrella — the compile-from-spec model, provenance header, run/activation path, and the explicit scope boundary against detector auto-activation (G-seed) and the S5 anti-hand-edit gate.

> **Stage:** S4 of generator-forbid-mvp (depends on S2 #689 + S3 #693, both merged to staging). Acceptance source: umbrella kickoff §2/§4.

## §1 Goal & scope

Turn the lookup step into a **compilation** step: a `declarative` `forbid` spec is compiled by the generator into ESLint-rule data + a generated `RULES.md` fragment + a provenance header, with **no handwritten rule file and no in-box markdown template per forbid**. This is the MVP headline — "a generator of rules-as-tests, not a library of fitness functions".

**In scope (S4-MVP, closes acceptance kickoff §4):**

1. Compile `rulesMd` from the declarative spec (drop the per-recipe handwritten `rulesMdTemplate` for declarative).
2. Emit a provenance header (generated-marker + source + content-hash).
3. Prove the compile end-to-end via a `--from-research` run + a validator fixture carrying a declarative forbid — activating S3 gates 7-8 (`n/a → pass`).
4. Calibrate `MAX_TOKEN_EDITS` against the first real pair (measured = 1).

**Out of scope (explicit boundaries — T4 premature-close counter):**

- **Detector auto-activation is OUT (G-seed).** Making a real `detectStack → research → synthesize` run emit the declarative forbid would require a new entry in `detector/patterns.ts` `KNOWN_PATTERNS` + `detectPatterns` — that is catalog/detection growth (G-seed, umbrella §3 scope fence) AND a line of handwritten code per forbid class (a soft T-MVP-A violation). S4 proves compilation via `--from-research`; auto-activation is deferred. See §2.4.
- **S5 anti-hand-edit gate is OUT.** S4 emits the provenance header; the mechanical CI rejection of a hand-edited generated file is S5 (the next stage).
- **require/type-aware codegen (G3b), ast-grep runner (decision (i)), polyglot tier** — all OUT per umbrella §3.

## §2 Forks resolved (decided on the merits, not by punting)

### §2.1 F1 — data-spec model → recipe-data, drop the template

**Decision:** a declarative forbid stays a recipe-JSON `data` shape (`selector` + `message` + `examples` + `title`); the generator compiles `rulesMd` + provenance + ESLint config from that data. The handwritten `rulesMdTemplate` field is ignored for `check.type === "declarative"`.

**Why determinate (not taste):** umbrella §3 scope fence forbids changing the `synthesize.ts` input surface beyond forbid-compilation, ruling out a separate `forbid-specs/*.json` channel or rewriting the `research → synthesize` contract. The S1a anchor table ([`s1a-eslint-forbid-pilot.md:97,104`](s1a-eslint-forbid-pilot.md)) already fixed S4 as "compilation step: selector → eslintConfigSnippet" on existing schema fields. T-MVP-A is satisfied: a new forbid is added as data, zero handwritten markdown/code.

### §2.2 F2 — provenance → generated-marker + source + content-hash

**Decision:** the emitted artifacts carry a provenance header with: (a) a `generated by rules-as-tests-synth — do not edit` marker, (b) the source (`patternId` + research `provenance`), (c) a `content-hash` = SHA-256 over the canonical JSON of the rule's identity-bearing fields (`check` + `examples` + `title`, key-sorted). S4 emits the header; S5 adds the CI check that recomputes the hash and reddens on mismatch.

**Why content-hash over whole-file-hash:** hashing the canonical rule content (not raw file bytes) is stable against reformatting/whitespace, and still detects a semantic hand-edit. Exact CI comparison semantics belong to S5 (out of scope here); S4 only lays the deterministic field.

### §2.3 F3 — fixtures → stay in the emitted manifest

**Decision:** paired fixtures remain `examples.{good,bad}` inside `rules-manifest-additions.json` (already emitted via `emit.ts:26-29`); S4 does NOT write separate `.good.ts`/`.bad.ts` files.

**Why determinate:** S1a anchor #3 ([`s1a-eslint-forbid-pilot.md:98`](s1a-eslint-forbid-pilot.md)) says "existing schema fields — no change needed"; `gate-rule-tester` runs examples **in-memory** via `Linter.verify` (not from files). Separate fixture files are YAGNI.

### §2.4 Detector activation boundary (the corrected pencil-mark)

`loadEntries` (`research/load.ts:76`) loads a store entry **only for an id the detector returned** in `detection.patterns`; `detectPatterns` (`detector/patterns.ts:79-103`) is a fixed 5-pattern list with no `test-only-forbid`. So a store entry alone does NOT auto-activate the forbid. S4 therefore proves compilation by **feeding a `ResearchPlan` directly via `--from-research`** (`synthesizer/cli.ts:43,58-60`) + a validator fixture — not by editing the detector. Auto-activation = G-seed, OUT (§1).

## §3 Design

### §3.1 Compile `rulesMd` (component 1)

A pure helper (e.g. `compile-declarative-md.ts`) maps a declarative `SynthesizedRule` → a markdown fragment deterministically from `title`, `check.selector`, `check.message`, `check.presence`. In `synthesize.ts`, for `check.type === "declarative"` the generated fragment is used **instead of** `recipe.rulesMdTemplate`. Non-declarative recipes keep `rulesMdTemplate` unchanged (no regression to the 5 existing check types). The `test-only-forbid-declarative.json` recipe's `rulesMdTemplate` becomes dead for the declarative path and is removed (data-only recipe).

### §3.2 Provenance header (component 2)

`emit.ts` gains a header block prepended to the emitted artifacts (or a sibling `provenance.json` for the machine-readable hash + a comment header in `RULES-additions.md`). Header carries the §2.2 fields. The hash function is a small pure util (`canonical-rule-hash.ts`) so S5 can import the identical function — single source of truth (dual-implementation-discipline §7: one logic, the S5 gate points at it via `spec:` marker).

### §3.3 Run / activation (component 3)

- New fixture: a `ResearchPlan` JSON containing the `test-only-forbid-declarative` pattern entry, fed via `--from-research`.
- The synth output (a `SynthesisPlan` with one declarative rule) feeds `validate()` → gates 7-8 go `n/a → pass` (the rule is present, minimal, messageId-covered, autofix-clean).
- A validator fixture + expected snapshot captures the `pass` outcome (mirrors `expected-fixture-validate.json`, which today shows the three S3 gates as `n/a`).

### §3.4 Gate calibration (component 4)

`gate-single-token-diff.ts:9-15,29` documents `MAX_TOKEN_EDITS = 5` as a "conservative interim, calibrated at S4". S4 records the calibration: the first real pair (`it.only(…)` vs `it(…)`) measures **token-distance = 1** — confirming real declarative pairs are ≈1 token (the umbrella §S3 ideal), so the threshold 5 stays a conservative-safe upper bound. Comment updated from "no real pair exists yet" to the measured fact.

## §5 Self-application (T15) + active traps

- **T15:** the generated declarative rule sits under principle 02 (paired good/bad) + 03 (AST-over-grep) — the generator holds itself to the same gate it ships to consumers.
- **T-MVP-A** (disguised lookup): a new forbid is added as recipe **data** only (selector+message+examples+title), zero handwritten code/template. The dead `rulesMdTemplate` removal is the proof the template is no longer load-bearing.
- **T4** (premature close): detector auto-activation explicitly named OUT (§2.4) rather than silently skipped.
- **T2/T3** (designing ≠ running): every "compiles"/"gate passes" claim in the PR carries a command + output (the `--from-research` run + `validate` output).

## §6 Capability-commit discipline

S4 touches `packages/core/**` (≥80 LOC likely) → capability commit. `Prior-art:` trailer with SSOT consult + DeepWiki/WebSearch ≥3 phrasings on "declarative ESLint-rule compilation from a spec" / "generated-file provenance marker / generated-by hash". ESLint tier = zero new deps (in-stack). No new dependency in S4.
