## Principle 2 ‚Äî Paired negative test

**Source:** [overview.md](../../skills/rules-as-tests/references/overview.md) Layer 3
  anti-pattern ("Expected value generated by running the code ‚Üí tautology farm") +
  [self-testing-docs.md](../../skills/rules-as-tests/references/self-testing-docs.md)
  ("Probe –±–µ–∑ negative test –Ω–µ —Å—á–∏—Ç–∞–µ—Ç—Å—è —Ä–µ–∞–ª–∏–∑–æ–≤–∞–Ω–Ω—ã–º")

**Test file:** [scripts/principles/02-paired-negative-test.test.ts](../../scripts/principles/02-paired-negative-test.test.ts)

**Manifest fields checked:** `examples.bad` (‚â•5 chars, non-trivial) and `examples.good`
  (‚â•5 chars); `examples.bad !== examples.good`.

**Pass criterion:** Every rule has distinct bad/good examples of non-trivial length.
  The negative example (`bad`) documents what violates the rule; without it, there is no
  documented violation scenario ‚Äî the rule enforcement is unverifiable by inspection.

**Mutation test:** sets `examples.bad = "   "` ‚Üí throws "absent or too short".
  Second mutation: sets `examples.bad === examples.good` ‚Üí throws "tautology".

**Currently passing:** 26/26 rules

**Note on Phase 5 scope:** Phase 2 covers STRUCTURAL presence. Actually running `examples.bad`
  through the ESLint rule and verifying it produces a lint error ‚Üí Phase 5 (validator runtime).

## Principle 3 ‚Äî AST > grep (structural, PARTIAL)

**Source:** [overview.md](../../skills/rules-as-tests/references/overview.md) Layer 2
  ("AST scan" patterns) + [SKILL.md](../../skills/rules-as-tests/SKILL.md)
  ("Meta-tests ‚Äî Implemented as AST scans")

**Test file:** [scripts/principles/03-ast-over-grep.test.ts](../../scripts/principles/03-ast-over-grep.test.ts)

**Manifest fields checked:** for rules with `check.type === "eslint"`: the `rule` field
  must be a non-empty, non-placeholder string referencing a concrete ESLint rule name.

**Pass criterion:** ESLint type rules reference concrete, non-placeholder rule names.
  ESLint rules use TSESTree/espree AST by default ‚Äî asserting concrete rule name is the
  structural proxy for "AST-based enforcement is in place."

**Applicability:** 11/26 rules have `check.type === "eslint"`. Rules of type
  "command"/"script"/"manual" are exempt ‚Äî they use other enforcement mechanisms.

**Mutation test:** sets `rule = ""` ‚Üí throws "rule name is empty".
  Second mutation: sets `rule = "TODO: add rule name"` ‚Üí throws "placeholder".

**Currently passing:** 11/11 eslint-type rules

**Known limitation (Phase 5 scope):** verifying that the ESLint rule's INTERNAL implementation
  uses AST traversal rather than regex requires running the rule. That is Phase 5 (validator
  runtime). Phase 2 covers only structural: "a concrete rule name exists."

## Principle 4 ‚Äî No tautology

**Source:** [ai-traps.md](../../skills/rules-as-tests/references/ai-traps.md) ¬ß3
  ("Tautological tests") + [overview.md](../../skills/rules-as-tests/references/overview.md)
  Layer 3 anti-pattern ("Expected value generated by running the code ‚Üí tautology farm")

**Test file:** [scripts/principles/04-no-tautology.test.ts](../../scripts/principles/04-no-tautology.test.ts)

**Manifest fields checked:**
  1. `check.command` must not be a no-op (`true`, `echo ok`, `:`, `exit 0`)
  2. `policy` must not be identical to `title`
  3. `examples.bad !== examples.good` (cross-check with P2)

**Pass criterion:** No rule has a structurally tautological manifest entry.

**Mutation tests:**
  - Sets `check.command = "true"` ‚Üí throws "no-op"
  - Sets `policy = rule.title` ‚Üí throws "policy field is identical to title"
  - Sets `examples.bad = examples.good` ‚Üí throws "tautological example pair"

**Currently passing:** 26/26 rules

**Note on Guardrail 2:** This principle is the meta-level self-application of Guardrail 2.
  The mutation tests for principles 1-7 themselves verify that each meta-test is not
  tautological ‚Äî they would fail if the assertions were trivially always-passing.

## Principle 6 ‚Äî MUST not demoted to should

**Source:** [ai-traps.md](../../skills/rules-as-tests/references/ai-traps.md) ¬ß6
  ("AI agent silently demotes 'MUST' to 'should'")

**Test file:** [scripts/principles/06-must-not-demoted.test.ts](../../scripts/principles/06-must-not-demoted.test.ts)

**Manifest fields checked:** `policy` field of automated-check rules (`check.type` ‚âÝ "manual")
  scanned for soft-only obligation language without hard counterpart.

**Applicability:** 21/26 rules have automated checks (not manual). Manual rules exempt ‚Äî
  their enforcement is inherently advisory.

**Soft patterns detected:** `consider`, `should consider`, `if possible`, `where possible`,
  `recommended`, `ideally`, `preferably`.

**Hard counterparts (exempting soft):** `MUST`, `must not`, `forbidden`, `prohibited`,
  `required`, `No `, `never`, `always`, `only`.

**Pass criterion:** Automated-check rules with policy fields either (a) use hard obligation
  language, or (b) contain BOTH soft and hard language (soft word in context of hard rule
  is acceptable).

**Mutation tests:**
  - Sets policy = "Consider using safer alternatives where possible. Recommended if feasible."
    (no hard language) ‚Üí throws "soft-obligation language"
  - Sets policy = "MUST use Zod. Recommended to also add logging where possible."
    (both) ‚Üí does NOT throw (correct: hard counterpart exempts soft)

**Currently passing:** 26/26 rules (21 automated-check rules all use hard language)

## Principle 7 ‚Äî Documents lie (examples are code, not prose)

**Source:** [self-testing-docs.md](../../skills/rules-as-tests/references/self-testing-docs.md)
  ("Documents lie; tests don't") + [SKILL.md](../../skills/rules-as-tests/SKILL.md)
  intro + [overview.md](../../skills/rules-as-tests/references/overview.md) Layer 5

**Test file:** [scripts/principles/07-documents-lie.test.ts](../../scripts/principles/07-documents-lie.test.ts)

**Manifest fields checked:** `examples.bad` and `examples.good` must contain code-like
  tokens (brackets, operators, string literals, JS/TS keywords, YAML structures, method
  calls) rather than pure prose descriptions.

**Pass criterion:** Examples contain executable code (any recognized code signal token),
  not just prose like "this is wrong because it violates the rule."

**Code signal patterns:** brackets `(){}[]`, operators `;,=`, string literals, JS/TS
  comments `//`, keywords (`import`/`export`/`const`/etc.), method calls `.foo(`,
  YAML patterns (`# `, `- `, `key: `), GitHub Actions keywords, `@version` refs.

**Mutation tests:**
  - Sets `examples.bad = ""` ‚Üí throws "examples.bad is empty"
  - Sets `examples.bad = "This is wrong because it violates the rule"` ‚Üí throws "prose"
  - Sets `examples.good = "This is the correct approach following best practices"` ‚Üí throws "prose"

**Currently passing:** 26/26 rules

**Known limitation (Phase 5 scope):** Actually running `examples.bad` through the ESLint
  rule to verify it produces a lint finding (and `examples.good` produces none) ‚Üí Phase 5.
  Phase 2 covers only structural: "example looks like code, not prose."
